How does Atlas handle ambiguous user input when multiple topics could match?CONCEPT
  • Confidence scoring — Atlas scores each topic based on match quality between user message and topic description + instructions
  • If top score is high → routes directly to that topic
  • If multiple topics score similarly → Atlas generates a clarifying question
  • If no topic exceeds threshold → falls back to system prompt default (polite decline + escalation offer)
  • Negative instructions remove ambiguity: "This topic does NOT handle payment disputes"
Critical insight

The quality of your topic descriptions is MORE important than prompt templates. Atlas routes BEFORE the LLM fires.

Explain the full Atlas reasoning loop with a real multi-step mortgage example.SCENARIO

User: "I want to know my loan balance and also set up autopay"

TURN 1: User → "I want my loan balance and set up autopay" REASON #1: Two intents detected. Start with loan balance (read-only, safer). Match → Topic: "Loan Inquiry" Plan → Call action: getLoanBalance(customerId) ACT #1: Execute getLoanBalance → Returns: {balance: $243,500, rate: 6.5%} OBSERVE: Got loan data. First intent resolved. Now address autopay. REASON #2: Second intent = autopay setup. Match → Topic: "Payment Management" Need bank account info → Ask user first. ACT #2: "Your balance is $243,500 at 6.5%. Which bank account for autopay?" TURN 2: User → "Use Chase checking ending in 4521" REASON #3: Bank info provided. Execute setup. Plan → Call action: setupAutoPay(userId, acct: 4521) ACT #3: Execute → {success: true, startDate: "June 1"} ACT #4: "Done! Autopay set up with Chase ****4521, starting June 1."
Key insight

Atlas decomposes multi-intent messages and addresses them sequentially, maintaining state across the loop.

How does Atlas handle conversation context and memory? What are the limits?CONCEPT
  • Within a conversation: Atlas maintains the full transcript — every message, response, and action result accumulated as context
  • Token limit: Long conversations may hit context window constraints — older turns get summarized or truncated
  • No cross-conversation memory: Each new session starts fresh. Agent does NOT remember past conversations
  • Workaround: Store preferences in CRM fields (language, communication prefs) and retrieve via grounding at session start
  • Session variables: Within one conversation, topic state carries information between actions
Key phrasing

"Agentforce has session memory, not persistent memory. For persistent personalization, use Data Cloud unified profiles."

What is "Plan" mode vs "Act" mode in Atlas?ARCHITECTURE
  • Plan mode — Atlas generates a step-by-step plan and shows to user for approval BEFORE executing. For high-risk operations
  • Act mode — Atlas executes actions immediately as it reasons. Faster, but less user control
  • Hybrid: Configure per-topic — read-only = Act mode (speed), transactional = Plan mode + confirmation (safety)
  • Mortgage: Balance inquiry = Act mode. Payoff request = Plan mode with explicit confirmation
What is Agentforce 2.0's multi-agent orchestration? How does it work?CONCEPT
  • Multi-Agent Orchestration — orchestrator agent delegates sub-tasks to specialist agents. Like a manager routing to the right team member
  • Example: Orchestrator receives "I want to refinance and update my address" → delegates to Refinance Agent AND Profile Update Agent
  • MuleSoft native integration — agents browse and invoke MuleSoft API catalog directly
  • Slack Agents — deployable into Slack for internal use cases (IT helpdesk, HR onboarding)
  • Tableau integration — agents query Tableau analytics and include chart summaries
