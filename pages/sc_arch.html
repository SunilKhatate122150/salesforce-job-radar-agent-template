//    YES + small data volumes → USE FLOW (Record-Triggered)

// 2. Does it involve bulk operations (200+ records at once)?

//    YES → USE APEX (Queueable from trigger, or @InvocableMethod from Flow)

// 4. Is performance critical?
//    YES → USE APEX (triggers are faster than Flow for bulk operations)

// 5. Is it Process Builder or Workflow?
//    ALWAYS MIGRATE TO FLOW — both are deprecated/retiring

// NEVER use:
// ❌ Process Builder for new logic — deprecated, replacing with Flow
How do you approach performance testing before a large Salesforce deployment?ARCHITECTURE▼
  • Query Plan analysis: Run all SOQL from triggers and LWC wire calls through Developer Console with EXPLAIN. Fix any TableScan before deployment
  • Bulk trigger test at 200 records: ALWAYS test triggers inserting exactly 200 records. If test only uses 1 record, governor limit issues will not surface until production
  • Batch at scale in Full Sandbox: Full Sandbox = copy of production data. Run batch with scope=1, then scope=50, then scope=200. Monitor governor limit consumption at each step
  • LWC with realistic data: Test datatable components with the maximum realistic row count (100, 500, 1000 rows). A datatable with 1000 rows renders very differently than one with 10
  • Limits.getCpuTime() profiling: Add CPU time checks at key points in complex Apex to identify bottlenecks before they hit the 10,000ms limit in production
  • Debug log analysis: Enable debug logs during load tests. Look for: slow queries (>200ms), repeated queries inside loops, unexpected governor consumption
// CPU time profiling in Apex
Long startCpu = Limits.getCpuTime();
// ... some operation ...
Long elapsed = Limits.getCpuTime() - startCpu;
System.debug('CPU used for operation: ' + elapsed + 'ms of ' + Limits.getLimitCpuTime() + 'ms total');
How do you manage technical debt in an existing Salesforce org with outdated automation?ARCHITECTURE▼
  • Audit with Salesforce Optimizer: Run the free Salesforce Optimizer report (Setup → Optimizer). It identifies unused fields, inactive flows, low-coverage Apex, duplicate rules, excessive objects
  • Migrate Process Builder and Workflow to Flow: Both are deprecated. Salesforce retirement dates are announced — start migration before forced retirement causes failures
  • Refactor triggers to handler pattern: Any org with logic directly in .trigger files (not handler classes) is a debt item. Prioritize the most-changed objects first
  • Coverage baseline: Any Apex class below 75% coverage = untested code that will fail. Treat each low-coverage class as a debt bug to fix
  • Dead code removal: Use "Where is this used?" for custom fields. Fields used in 0 automations, 0 layouts, 0 reports can be candidates for removal (verify with business first)
  • Tech debt as backlog items: Frame debt in business terms — "Refactoring Account trigger reduces deployment time from 4 hours to 45 minutes due to eliminated merge conflicts"
  • Dedicated cleanup time: Reserve 10–20% of each sprint for debt reduction. Without dedicated time, debt only accumulates
How do you stay current with Salesforce releases as a senior developer?ARCHITECTURE▼

Salesforce releases 3 major updates per year: Spring (February), Summer (June), Winter (October/November).

  • Release Notes: Read for every release. Focus on: Apex API changes, LWC new capabilities, Flow enhancements, deprecated features and their timelines
  • Pre-release orgs: Salesforce makes preview orgs available 4–6 weeks before each release. Test existing code against the new release before it hits production
  • Trailhead Release modules: 1–2 new Trailhead modules released with each major release — completing them is the fastest way to learn what changed
  • Certification maintenance: Salesforce certs require annual maintenance exams for each release. These force you to learn what changed — make them part of your professional development
  • Salesforce Developer Blog: Official technical announcements. Follow on Trailblazer Community. Subscribe to the developer newsletter
  • Deprecation tracking: Keep a personal deprecation timeline — when does Process Builder retire? When does With SECURITY_ENFORCED become the wrong choice? Plan migrations proactively
Interview answer

Say: "I stay current through the official release notes and Salesforce Developer Blog, and I test major features in sandbox before they reach production. I also track the Process Builder and Workflow deprecation timeline because we have existing automations that need migration to Flow — I want to stay ahead of forced retirements rather than scramble at the deadline."

Design a real-time loan status notification system from scratch — architect the complete solutionARCHITECTURE▼

Requirements: send email, SMS, and push notifications when loan status changes. Multiple loan types. Configurable rules per status change.

Architecture decision — Why NOT direct in trigger: The trigger would need to know about every notification channel, every loan type's rules, and every recipient configuration. That is too tightly coupled.

Decoupled design:

  • Trigger: After update on Loan__c — if Status__c changed → publish Loan_Status_Changed__e Platform Event (loanId, oldStatus, newStatus). Nothing else in the trigger.
  • Configuration: Notification_Config__mdt (Custom Metadata) — rows defining: LoanType, FromStatus, ToStatus, Channels (Email/SMS/Push), RecipientType (Borrower/LO/Manager). No code change needed to add/change rules.
  • Router class: Platform Event trigger subscribes to Loan_Status_Changed__e. Queries Notification_Config__mdt for matching rules. For each matching rule, calls the appropriate channel adapter via INotificationChannel interface.
  • Channel adapters: EmailAdapter (Salesforce Messaging), SMSAdapter (Twilio via Named Credential callout), PushAdapter (Salesforce Notification Builder).
  • Error handling: Every notification attempt logged to Notification_Log__c. Failed sends retried by scheduled Apex up to 3 times.
Patterns demonstrated

Platform Events (decoupling), Custom Metadata (configuration-driven, deploy without code change), Strategy Pattern (INotificationChannel), Error logging, Retry logic. This answer shows senior architect thinking — decoupling, configuration-driven design, error handling, retry logic.
