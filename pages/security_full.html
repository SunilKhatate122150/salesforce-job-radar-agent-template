Object, Field, and Record level security — declarative vs programmatic enforcement
Explain the 3 layers of Salesforce Security in 30 seconds — "CRUD, FLS, and Sharing".SECURITY

Interviewers expect you to break it down exactly like this:

  • Layer 1: Object Level (CRUD) — Profiles and Permission Sets. Determines if a user can Create, Read, Edit, or Delete records of a specific object. (e.g., Can this user even see the Loan object?)
  • Layer 2: Field Level (FLS) — Profiles and Permission Sets. Determines if a user can see or edit specific fields on an object. (e.g., Can they see the SSN field on the Loan object?)
  • Layer 3: Record Level (Sharing) — OWD, Role Hierarchy, Sharing Rules, Apex Sharing. Determines WHICH specific records of that object the user can see. (e.g., I can see Loans, but can I see ONLY MY loans or ALL loans in the org?)
Memory Hook

CRUD/FLS is "What can I do?" | Sharing is "Which ones can I do it to?"

Declarative Sharing Model: OWD vs Role Hierarchy vs Sharing Rules — give a mortgage scenario.SECURITY

Sharing is always additive — you start with the most restrictive baseline (OWD) and open up access.

  • OWD (Organization Wide Defaults): The baseline. Set to Private for Loans. Only the owner and admins see their own records.
  • Role Hierarchy: Vertical opening. A Regional Manager (higher in hierarchy) automatically sees all records owned by the Loan Officers below them.
  • Sharing Rules (Horizontal): Opening across roles. Rule: "Share all Loans owned by Branch A with the Compliance Team role."
  • Manual Sharing: One-off opening. "Loan Officer A manually shares their loan record with Underwriter B for review."

What is "with sharing" vs "without sharing" vs "inherited sharing" in Apex — when to use each?SECURITY
// with sharing — Respects the user's sharing rules (Record level)
// If the user can't see the record via sharing rules, SOQL returns 0 results.
public with sharing class LoanService {
    public static List<Loan__c> getMyLoans() {
        // only returns records user has access to
    }
}

// without sharing — Runs in "System Mode" — Ignores sharing rules
// User sees ALL records in the org regardless of sharing rules.
// Use for: Batch jobs, background processors, system validation.
public without sharing class SystemManager { ... }

// inherited sharing — RECOMMENDED for utility/service classes
// Takes the sharing mode of the class that called it.
public inherited sharing class LoanUtils { ... }
Security Alert

Apex classes DO NOT enforce CRUD or FLS by default, even if you use "with sharing". "with sharing" ONLY enforces record-level sharing. You must explicitly enforce CRUD/FLS in code.

What is Apex Managed Sharing — when and how to implement it?SECURITY

Used when declarative sharing (Role Hierarchy, Sharing Rules) isn't complex enough. Example: "A loan should be shared with an Underwriter ONLY IF their credit specialty matches the loan type."

// Create a Sharing record (the __Share object)
Loan__Share lShare = new Loan__Share();
lShare.ParentId = 'a00...'; // The record being shared
lShare.UserOrGroupId = '005...'; // The user getting access
lShare.AccessLevel = 'Read'; // 'Read', 'Edit', 'All'
lShare.RowCause = Schema.Loan__Share.RowCause.Specialty_Match__c; // Optional: reason

insert lShare;
  • Custom Row Cause: Create in Setup so you can track WHY the record was shared.
  • Standard vs Custom: For standard objects (AccountShare), the field is AccountId. For custom (Loan__Share), it is ParentId.
  • Deletion: If you manually insert share records, you must manually delete them if the condition changes (unless using a standard RowCause).
What is the "Guest User Security Policy" — how does it affect Experience Cloud sites?SECURITY
  • Private by Default: All objects are Private for Guest users. You cannot make them Public.
  • Sharing Rules: You must use Guest User Sharing Rules (Criteria-Based) to give access to guests.
  • Ownership: Guest users cannot OWN records. New records created by guests must be assigned to a default internal user.
  • CRUD: Guests can never have "Update" or "Delete" permissions. They can only "Create" and "Read" (if sharing rules allow).
Experience Cloud Tip

If a guest user needs to update a record (e.g., submitting a loan application update), the LWC must call a "without sharing" Apex class to perform the update on their behalf securely.

What are Permission Set Groups and Muting Permission Sets — why use them?SECURITY

Modern Salesforce security favors "Minimum Viable Profile" + Permission Sets.

  • Permission Set Group: Bundles multiple permission sets together. Assign one group to a user instead of 10 individual sets. (e.g., "Senior Underwriter Group" = Base Underwriter + Credit Approval + Manager Overrides).
  • Muting Permission Set: Used inside a group to REJECT a specific permission that one of the bundled sets provides. This allows for granular control without creating dozens of slightly different permission sets.
