How does @InvocableMethod work differently in Agentforce vs Flow?CODE▼

Every Apex agent action IS an @InvocableMethod — you already know this pattern. But in Agentforce: agent calls it dynamically based on reasoning, inputs are populated by Atlas from conversation context, and output is used for next reasoning step.

CRITICAL: description on @InvocableVariable is cosmetic in Flows but functional in Agentforce — Atlas reads it to understand parameters.

public class LoanStatusAction {
    public class Request {
        @InvocableVariable(
            required=true 
            description='The Salesforce ID of the customer loan record'
        )
        public String loanId;
    }
    
    public class Response {
        @InvocableVariable(description='Current outstanding loan balance in USD')
        public Decimal balance;
        
        @InvocableVariable(description='Whether the action completed successfully')
        public Boolean success;
        
        @InvocableVariable(description='Error message if action failed')
        public String errorMessage;
    }
    
    @InvocableMethod(
        label='Get Loan Status' 
        description='Retrieves current balance and status for a mortgage loan'
    )
    public static List<Response> execute(List<Request> requests) {
        // Your existing Apex patterns apply 1:1 here
    }
}
Key insight

"Descriptions aren't just documentation — they're instructions to the AI reasoning engine."

How should you handle errors in Agentforce Apex actions?CODE▼

NEVER let actions throw raw exceptions — return structured error responses instead.

@InvocableMethod(label='Get Loan Balance')
public static List<Response> getLoanBalance(List<Request> requests) {
    List<Response> responses = new List<Response>();
    for (Request req : requests) {
        Response res = new Response();
        try {
            Loan__c loan = [SELECT Balance__c, Status__c 
                           FROM Loan__c WHERE Id = :req.loanId LIMIT 1];
            res.success = true;
            res.currentBalance = loan.Balance__c;
        } catch (QueryException e) {
            res.success = false;
            res.errorMessage = 'Unable to retrieve loan details. ' +
                'Let me connect you with a team member.';
        }
        responses.add(res);
    }
    return responses;
}
UX principle

"Graceful errors in agent actions are critical — a raw exception becomes a confusing 'Something went wrong' to the customer."

Agent actions must be synchronous — how do you handle long-running processes?CODE▼
  • Actions execute within reasoning loop — must return synchronously
  • But an action can ENQUEUE a Queueable for downstream processing
  • Agent confirms doc upload → Apex saves doc → enqueues Queueable for OCR → responds immediately
  • Agent creates payoff request → saves record + enqueues Queueable callout to LOS → tells user "processing..."
Your advantage

"The action is the front door — fast and responsive. The Queueable is the back office — heavy lifting happens async."

Your trigger framework — how does it interact with Agentforce?CODE▼
  • Agent actions create/update CRM records → your triggers WILL fire
  • Critical: If action creates a Case and you have a before-insert trigger → runs in action's execution context → same governor limits
  • Recursion risk: Action creates record → trigger fires → trigger updates → trigger fires again. Your recursion guard prevents this
  • Design awareness: Must understand full order-of-execution including existing triggers, flows, validation rules
Competitive edge

"This is where my Apex experience gives me a real advantage — order of execution, recursion prevention, governor limits from day one."

How can LWC be used alongside Agentforce?CODE▼
  • Embedded agent: lightning-messaging-utility component embeds agent chat in any LWC page
  • Custom rich output: Agent returns structured data → your LWC renders it as a loan dashboard or payment timeline
  • Agent-assisted forms: Loan application LWC with "Ask Agent" button → opens contextual conversation

Agent Testing Framework — how do you write tests?ARCHITECTURE▼
  • Conversation-level: Agent Conversation Simulator to replay conversations, verify topic routing + action invocation
  • Action-level: Each Apex action = @InvocableMethod → write standard Apex test classes
  • Prompt Template: Preview in Prompt Builder with sample record IDs
  • Regression: Bank of test conversations (happy path + edge cases) re-run after every instruction change
  • Metrics: Topic match accuracy, action completion rate, escalation rate, avg turns to resolution
@isTest
public class LoanStatusActionTest {
    @isTest
    static void testGetLoanStatus() {
        Loan__c loan = new Loan__c(Balance__c=250000, Rate__c=6.5, Status__c='Active');
        insert loan;
        
        List<LoanStatusAction.Request> reqs = new List<LoanStatusAction.Request>();
        LoanStatusAction.Request req = new LoanStatusAction.Request();
        req.loanId = loan.Id;
        reqs.add(req);
        
        Test.startTest();
        List<LoanStatusAction.Response> resps = LoanStatusAction.execute(reqs);
        Test.stopTest();
        
        System.assertEquals(250000, resps[0].balance);
        System.assert(resps[0].success);
    }
