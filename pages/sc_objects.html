// Mortgage example: Lookup — Contact to Account
// Contact can exist without Account (orphan contact is valid)
Contact c = [SELECT Id, AccountId FROM Contact WHERE AccountId = null]; // valid query

// Master-Detail — Loan__c to Application__c
// Loan cannot exist without Application — Application__c field is required
// Rollup on Application: Total_Loan_Amount__c = SUM(Loan__c.LoanAmount__c)
Loan__c orphanLoan = new Loan__c(LoanAmount__c=100000);
// insert orphanLoan;  // FAILS — Application__c is required (master-detail)

// Delete cascade — delete Application deletes ALL its Loans automatically
delete [SELECT Id FROM Application__c WHERE Id = :appId];
// All child Loan__c records are now deleted too

Say: "In my Mortgage Platform, Loan__c had a Master-Detail to Application__c because a loan cannot logically exist without an application, and we needed a Rollup Summary to show total funded amount on the Application record. For the Contact-to-Account relationship we used Lookup because a loan officer (Contact) could exist in the system before being assigned to a broker office (Account)."

How do you implement a many-to-many relationship using a Junction Object — build one for the mortgage domain?OBJECTS▼

Salesforce does not support many-to-many natively. Use a Junction Object — a custom object with two Master-Detail fields pointing to each parent.

// Scenario: A Loan can have multiple Borrowers (Primary, Co-Borrower)
// A Contact (Borrower) can be on multiple Loans

// Junction Object: Loan_Borrower__c
// Field 1: Loan__c       — Master-Detail to Loan__c
// Field 2: Borrower__c   — Master-Detail to Contact
// Field 3: Role__c       — Picklist: Primary Borrower, Co-Borrower, Guarantor
// Field 4: CreditScore__c — Number (borrower's FICO at time of application)

// Query all borrowers on a loan:
List<Loan_Borrower__c> borrowers = [
    SELECT Borrower__r.Name, Borrower__r.Email, Role__c, CreditScore__c
    FROM Loan_Borrower__c
    WHERE Loan__c = :loanId
    ORDER BY Role__c
];

// Query all loans for a borrower (Contact):
List<Loan_Borrower__c> theirLoans = [
    SELECT Loan__r.Name, Loan__r.LoanAmount__c, Loan__r.Status__c, Role__c
    FROM Loan_Borrower__c
    WHERE Borrower__c = :contactId
    ORDER BY Loan__r.CreatedDate DESC
];

// Parent-to-Child subquery from Loan:
List<Loan__c> loansWithBorrowers = [
    SELECT Id, Name,
           (SELECT Borrower__r.Name, Role__c FROM Loan_Borrowers__r)
    FROM Loan__c WHERE Id = :loanId
];
What are External ID fields — when and how do you use them in integrations?OBJECTS▼

External ID fields store a unique identifier from an external system. Salesforce indexes them for fast lookups. They are critical for upsert operations in integrations — you can upsert by External ID instead of Salesforce Id.

// Field: Loan__c.LOS_Loan_ID__c (External ID, Unique, Text)
// Stores the ID from an external Loan Origination System

// UPSERT — matches on External ID, inserts if not found, updates if found
Loan__c loan = new Loan__c(
    LOS_Loan_ID__c = '2024-LOS-78542',
    LoanAmount__c = 350000,
    Status__c = 'Processing'
);
Database.upsert(loan, Loan__c.LOS_Loan_ID__c, false);
// false = partial mode — does not throw for single failures

// Bulk upsert from LOS integration:
List<Database.UpsertResult> results = Database.upsert(loansFromLOS, Loan__c.LOS_Loan_ID__c, false);

// REST API equivalent:
// PATCH /services/data/v59.0/sobjects/Loan__c/LOS_Loan_ID__c/2024-LOS-78542
// Body: {"LoanAmount__c": 350000, "Status__c": "Processing"}

// Best practices:
// Max 25 External ID fields per object
// Field types: Text, Number, Email, Auto Number only
// Mark as Unique to enforce no duplicates
// Do NOT use Id field as External ID — use a meaningful business identifier
Real use from your project

In the Mortgage Origination Platform, when syncing from the external LOS, we upserted on LOS_Loan_ID__c instead of Salesforce Id. This meant the integration never needed to query Salesforce first to find the record — just upsert with the LOS ID and Salesforce handled insert vs update automatically.

What is a Rollup Summary field — what are its limitations? How do you replicate it on a Lookup relationship?OBJECTS▼

Rollup Summary fields aggregate child record values (COUNT, SUM, MIN, MAX) onto a parent object. They only work on Master-Detail relationships.

  • Supported functions: COUNT, SUM, MIN, MAX
  • Can filter which child records to include (like a WHERE clause)
  • Update automatically when child records change — asynchronous on bulk loads
  • Cannot roll up formula fields or fields two levels away
  • Not available on Lookup relationships
// Replicating Rollup on a Lookup relationship using Apex trigger
// Scenario: Account (parent lookup) ← Contact (child)
// Goal: Count active contacts on Account.Active_Contact_Count__c

trigger ContactRollupTrigger on Contact (after insert, after update, after delete, after undelete) {

    Set<Id> accountIds = new Set<Id>();
    List<Contact> allContacts = Trigger.new != null ? Trigger.new : Trigger.old;
    for(Contact c : allContacts) {
        if(c.AccountId != null) accountIds.add(c.AccountId);
    }
    // Also include old AccountId on updates (contact moved to different account)
    if(Trigger.old != null) {
        for(Contact c : Trigger.old) {
            if(c.AccountId != null) accountIds.add(c.AccountId);
        }
    }
    if(accountIds.isEmpty()) return;

    List<AggregateResult> counts = [
        SELECT AccountId, COUNT(Id) cnt
        FROM Contact
        WHERE AccountId IN :accountIds AND IsActive__c = true
        GROUP BY AccountId
    ];

    Map<Id, Integer> countMap = new Map<Id, Integer>();
    for(AggregateResult ar : counts) countMap.put((Id)ar.get('AccountId'), (Integer)ar.get('cnt'));

    List<Account> toUpdate = new List<Account>();
    for(Id aId : accountIds) {
        toUpdate.add(new Account(Id=aId, Active_Contact_Count__c = countMap.containsKey(aId) ? countMap.get(aId) : 0));
    }
    update toUpdate;
}
How do field dependencies (controlling and dependent picklists) work — implement in LWC?OBJECTS▼

Field dependencies link two picklist fields. The Controlling field determines which values appear in the Dependent field. With lightning-record-edit-form, this works automatically. With custom lightning-combobox elements, you must filter manually in JavaScript.

// Custom LWC — filter dependent picklist manually
// Controlling: LoanType__c | Dependent: LoanProgram__c

handleLoanTypeChange(event) {
    this.selectedLoanType = event.detail.value;
    const programMap = {
        'Conventional': ['Conforming 30yr', 'Conforming 15yr', 'Jumbo'],
        'FHA': ['FHA 30yr', 'FHA 15yr', 'FHA Streamline'],
        'VA': ['VA 30yr', 'VA IRRRL', 'VA Jumbo'],
        'USDA': ['USDA Rural']
    };
    const programs = programMap[this.selectedLoanType] || [];
    this.loanProgramOptions = programs.map(p => ({ label: p, value: p }));
    this.selectedLoanProgram = '';
}

// Get picklist values from Salesforce metadata at runtime (best approach):
import { getPicklistValues, getObjectInfo } from 'lightning/uiObjectInfoApi';
import LOAN_PROGRAM from '@salesforce/schema/Loan__c.LoanProgram__c';

@wire(getObjectInfo, { objectApiName: 'Loan__c' }) objectInfo;

@wire(getPicklistValues, {
    recordTypeId: '$objectInfo.data.defaultRecordTypeId',
    fieldApiName: LOAN_PROGRAM
})
loanProgramPicklist({ data }) {
    // data.controllerValues maps controlling picklist value to index
    // data.values[i].validFor is a bitmask matching controller indices
    // Use this to filter dependent values based on selected controlling value
    this.allLoanPrograms = data;
}
How do you get all field metadata at runtime using Schema describe — use case: dynamic form builder?OBJECTS▼
// Get all fields and their properties at runtime
Map<String, Schema.SObjectField> fieldMap =
    Schema.SObjectType.Loan__c.fields.getMap();

List<Map<String,Object>> fieldData = new List<Map<String,Object>>();
for(String fieldName : fieldMap.keySet()) {
    Schema.DescribeFieldResult dfr = fieldMap.get(fieldName).getDescribe();
    if(!dfr.isAccessible()) continue; // skip inaccessible fields
    fieldData.add(new Map<String,Object>{
        'apiName' => dfr.getName(),
        'label' => dfr.getLabel(),
        'type' => String.valueOf(dfr.getType()),
        'required' => !dfr.isNillable() && dfr.isCreateable(),
        'updateable' => dfr.isUpdateable(),
        'length' => dfr.getLength()
    });
}

// Get picklist values at runtime — for dynamic dropdowns
List<Schema.PicklistEntry> statusVals = Loan__c.Status__c.getDescribe().getPicklistValues();
for(Schema.PicklistEntry pv : statusVals) {
    System.debug(pv.getValue() + ' — active: ' + pv.isActive() + ' default: ' + pv.isDefaultValue());
}

// Expose to LWC:
@AuraEnabled(cacheable=true)
public static List<Map<String,String>> getLoanStatusOptions() {
    List<Map<String,String>> opts = new List<Map<String,String>>();
    for(Schema.PicklistEntry e : Loan__c.Status__c.getDescribe().getPicklistValues()) {
        if(e.isActive()) opts.add(new Map<String,String>{'label'=>e.getLabel(),'value'=>e.getValue()});
    }
    return opts;
}
What are Record Types — when do you use them and how do you get the ID in Apex?OBJECTS▼

Record Types allow different page layouts, picklist values, and business processes per user type. In mortgage: Loan__c could have Record Types for Conventional, FHA, VA, USDA — each showing different fields and picklist values.

// Get Record Type Id — ALWAYS use DeveloperName, not Name
// Name can change (admin renames it), DeveloperName is stable
Id fhaRTId = Schema.SObjectType.Loan__c
    .getRecordTypeInfosByDeveloperName().get('FHA_Loan').getRecordTypeId();

Loan__c fhaLoan = new Loan__c(
    RecordTypeId = fhaRTId,
    LoanAmount__c = 300000
);

// Get all active record types for an object:
Map<String, Schema.RecordTypeInfo> rtMap =
    Schema.SObjectType.Loan__c.getRecordTypeInfosByDeveloperName();
for(String devName : rtMap.keySet()) {
    Schema.RecordTypeInfo rti = rtMap.get(devName);
    if(rti.isActive() && !rti.isMaster()) {
        System.debug(rti.getName() + ' → ' + rti.getRecordTypeId());
    }
}

// In LWC — use getObjectInfo wire adapter to get record types available to the user
import { getObjectInfo } from 'lightning/uiObjectInfoApi';
import LOAN_OBJECT from '@salesforce/schema/Loan__c';

@wire(getObjectInfo, { objectApiName: LOAN_OBJECT })
objectInfo;

get recordTypeOptions() {
    if(!this.objectInfo?.data) return [];
    return Object.values(this.objectInfo.data.recordTypeInfos)
        .filter(rt => rt.available && !rt.master)
        .map(rt => ({ label: rt.name, value: rt.recordTypeId }));
}
How do you implement a self-relationship (hierarchical data) in Salesforce — example with org chart?OBJECTS▼

A self-relationship is a lookup field on an object that points back to the same object. The Account.ParentId is the classic example. You can create custom self-lookups for hierarchical data like territory trees or category trees.

// Standard self-relationship: Account → ParentAccount
Account branch = [
    SELECT Id, Name, Parent.Name, Parent.Parent.Name
    FROM Account WHERE Id = :branchId
];
// Navigate up 5 levels max

// Custom self-relationship: Loan_Category__c → Parent_Category__c (Lookup to Loan_Category__c)
List<Loan_Category__c> hierarchy = [
    SELECT Id, Name, Parent_Category__r.Name, Parent_Category__r.Parent_Category__r.Name
    FROM Loan_Category__c
    WHERE Id = :categoryId
];

// Build full hierarchy path (all ancestors):
public static List<String> getCategoryPath(Id categoryId) {
    List<String> path = new List<String>();
    Id currentId = categoryId;
    while(currentId != null) {
        Loan_Category__c cat = [SELECT Id, Name, Parent_Category__c FROM Loan_Category__c WHERE Id = :currentId];
        path.add(0, cat.Name); // add at front for correct order
        currentId = cat.Parent_Category__c;
    }
    return path; // e.g. ['Real Estate', 'Residential', 'Single Family']
}
What are the different Salesforce ID formats — 15-digit vs 18-digit — when does it matter in code?OBJECTS▼

Salesforce IDs have two formats. Both refer to the same record — the difference is case sensitivity handling.

  • 15-character ID: Case-sensitive. Used in Salesforce UI URLs. Shorter. Comparison in Apex is case-sensitive for String comparison but case-insensitive when stored as Id type.
  • 18-character ID: Case-insensitive. Appends 3 checksum characters. Safe for use in Excel (which lowercases text), external systems, and case-insensitive string comparisons.
// Apex Id type handles both — comparison is always case-insensitive
Id id15 = '001Dn000009XYZa';
Id id18 = '001Dn000009XYZaIAG';
System.assertEquals(id15, id18); // TRUE — Apex Id type handles this

// Problem: String comparison is case-SENSITIVE
String strId = '001DN000009XYZA'; // different case
System.assertEquals((Id)strId, id15); // TRUE — cast to Id first

// Best practice: ALWAYS store and compare as Id type, not String
Id safeId = id15; // not String safeId = id15;

// Convert 15 to 18 digit in Apex:
String id18digit = (String) id15; // automatic when cast Id to String
System.debug(id18digit.length()); // 18

// Object prefix codes (first 3 chars) — useful for type checking:
// 001 = Account, 003 = Contact, 006 = Opportunity, 005 = User
// Custom objects vary by org — use Schema.getGlobalDescribe() to find them
String objType = Id.valueOf(someId).getSObjectType().getDescribe().getName();
How do you prevent a user from deleting a parent record if it has active child records — two approaches?OBJECTS▼
// APPROACH 1: Validation Rule on Parent
// Rule on Application__c object, fires on Delete:
// Condition: VLOOKUP("Loan__c Count", "Active_Loan_Count__c", Id) > 0
// (Requires a rollup summary field Active_Loan_Count__c on Application__c)
// Error: "Cannot delete Application with active Loans. Close or deny all loans first."

// APPROACH 2: Apex Before Delete Trigger (more flexible, checks any condition)
trigger ApplicationDeleteTrigger on Application__c (before delete) {
    Set<Id> appIds = Trigger.oldMap.keySet();

    // Check for active child loans
    Map<Id, Integer> activeLoans = new Map<Id, Integer>();
    for(AggregateResult ar : [
        SELECT Application__c, COUNT(Id) cnt
        FROM Loan__c
        WHERE Application__c AND Status__c 'Closed', 'Denied', 'Withdrawn')
        Id)ar.get('Application__c'), (Integer)ar.get('cnt'));
    }

    for(Application__c app : Trigger.old) {
        'Cannot delete Application "' + app.Name +
                '" — it has ' + activeLoans.get(app.Id) +
                ' active loan(s). Close or deny all loans first.'
            );
        }
    }
}
How does "Controlled by Parent" OWD work for Contact — explain the nuance?OBJECTS▼

"Controlled by Parent" is a special OWD setting for Contact. When set, Contact record access is determined by the parent Account's sharing. If a user can see the Account, they can see all its Contacts. If they cannot see the Account, they cannot see the Contacts.

  • Only available for Contact (standard) when Account is the parent
  • Contact inherits Account's OWD — if Account is Private, Contacts are effectively Private
  • If Account is Public Read Only → Contacts are Read Only to everyone
  • Sharing rules on Account automatically extend to its Contacts
  • Private Contacts (no AccountId) are only visible to the owner and admins regardless of this setting
  • This setting simplifies sharing model management — you only maintain Account sharing, not both Account and Contact separately
Practical implication

In a mortgage org: if Accounts (Broker Offices) are set to Private, and Contact uses "Controlled by Parent," then each loan officer only sees contacts at brokers they have access to. This prevents loan officer A from seeing loan officer B's borrower contacts — which is a FCRA/privacy best practice.
