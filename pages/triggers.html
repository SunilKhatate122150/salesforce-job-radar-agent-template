Walk me through your trigger handler pattern — show complete code TRIGGERS

The trigger handler pattern separates the trigger (routing) from the business logic (handler class). The trigger file should have ZERO business logic — it only calls the handler. The handler class has methods for each trigger event.

// STEP 1: The Trigger File (ContactTrigger.trigger)
// This file has NO logic - only routing
trigger ContactTrigger on Contact (
    before insert, before update, before delete,
    after insert, after update, after delete, after undelete
) {
    ContactTriggerHandler handler = new ContactTriggerHandler();
    handler.run();
}

// STEP 2: Base Handler Class (TriggerHandler.cls)
// Reusable base class for all triggers in the org
public virtual class TriggerHandler {
    
    private static Set<String> bypassedHandlers = new Set<String>();
    
    public void run() {
        if(isBypassed(getHandlerName())) return;
        
        switch on Trigger.operationType {
            when BEFORE_INSERT  { beforeInsert(); }
            when BEFORE_UPDATE  { beforeUpdate(); }
            when BEFORE_DELETE  { beforeDelete(); }
            when AFTER_INSERT   { afterInsert(); }
            when AFTER_UPDATE   { afterUpdate(); }
            when AFTER_DELETE   { afterDelete(); }
            when AFTER_UNDELETE { afterUndelete(); }
        }
    }
    
    protected virtual void beforeInsert() {}
    protected virtual void beforeUpdate() {}
    protected virtual void beforeDelete() {}
    protected virtual void afterInsert() {}
    protected virtual void afterUpdate() {}
    protected virtual void afterDelete() {}
    protected virtual void afterUndelete() {}
    
    public static void bypass(String handlerName) { bypassedHandlers.add(handlerName); }
    public static Boolean isBypassed(String handlerName) { return bypassedHandlers.contains(handlerName); }
    private String getHandlerName() { return String.valueOf(this).substring(0, String.valueOf(this).indexOf(':')); }
}

// STEP 3: Specific Handler (ContactTriggerHandler.cls)
public class ContactTriggerHandler extends TriggerHandler {
    
    private List<Contact> newContacts;
    private Map<Id, Contact> oldContactMap;
    
    public ContactTriggerHandler() {
        newContacts = Trigger.new;
        oldContactMap = (Map<Id, Contact>) Trigger.oldMap;
    }
    
    protected override void beforeInsert() {
        ContactService.preventDuplicates(newContacts);
        ContactService.populateDefaultValues(newContacts);
    }
    
    protected override void afterUpdate() {
        ContactService.syncPhoneToAccount(newContacts, oldContactMap);
    }
}
Interview answer tip

Explain the separation of concerns: trigger = routing only, handler = event dispatching, service = business logic. This makes code testable, maintainable, and allows bypass logic for data migrations. Always mention: "One trigger per object — all events in one file."

How do you prevent recursive triggers? What are the pitfalls? TRIGGERS

Recursive trigger = trigger fires, performs DML on same object, which fires the trigger again ΓåÆ infinite loop ΓåÆ CPU limit exceeded.

// Method 1: Static Boolean Flag (Simple but imperfect)
public class TriggerRecursionPrevention {
    public static Boolean isFirstRun = true;
}

trigger AccountTrigger on Account (after update) {
    if(TriggerRecursionPrevention.isFirstRun) {
        TriggerRecursionPrevention.isFirstRun = false;
        // logic here
    }
}
// PROBLEM: Only runs once per transaction even for different records

// Method 2: Set of processed IDs (Better - handles bulk correctly)
public class TriggerRecursionHelper {
    public static Set<Id> processedIds = new Set<Id>();
}

protected override void afterUpdate() {
    List<Contact> toProcess = new List<Contact>();
    for(Contact c : newContacts) {
        if(!TriggerRecursionHelper.processedIds.contains(c.Id)) {
            toProcess.add(c);
            TriggerRecursionHelper.processedIds.add(c.Id);
        }
    }
    if(!toProcess.isEmpty()) {
        ContactService.syncPhone(toProcess);
    }
}
Pitfall

Static variables persist for the entire transaction, not just one trigger call. In a batch job processing 200 records at a time, your static flag from chunk 1 still exists in chunk 2 — this can cause logic to be skipped entirely for later chunks. Design with this in mind.

// BEFORE INSERT — validate and set fields (no DML needed)
protected override void beforeInsert() {
    for(Loan__c loan : Trigger.new) {
        // Validation - field directly modifiable
        if(loan.LoanAmount__c > 2000000 && loan.LoanType__c == 'FHA') {
            loan.addError('FHA loans cannot exceed $2,000,000');
        }
        // Field default - set directly, no update DML needed
        if(loan.Status__c == null) loan.Status__c = 'Application';
    }
}

// AFTER INSERT — create related records (record now has Id)
protected override void afterInsert() {
    List<LoanTask__c> tasks = new List<LoanTask__c>();
    for(Loan__c loan : Trigger.new) {
        // Now we have loan.Id (not available in before insert)
        tasks.add(new LoanTask__c(Loan__c = loan.Id, Type__c = 'Initial Review'));
    }
    if(!tasks.isEmpty()) insert tasks;
}
