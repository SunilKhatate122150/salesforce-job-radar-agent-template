Flow vs Apex Trigger — the complete decision framework with examplesFLOW▼

Use Flow when:

  • Logic must be admin-maintainable without code deployment
  • Before Save field defaults or simple same-object updates (faster than After Save)
  • Sending email alerts, creating one related record (task, chatter post)
  • Calling an @InvocableMethod Apex action for a single step
  • Processing small record sets (under 50 records at once)

Use Apex Trigger when:

  • Bulk operations — triggers safely handle 200 records; Flow has governor limit risk at scale
  • Complex cross-object SOQL with multiple relationship levels
  • External API callouts required (Flow cannot make callouts natively)
  • Complex error handling — partial DML, retry logic
  • Performance-critical path — Apex triggers run faster than equivalent Flow for bulk
  • Logic that must be bullet-proof against governor limits regardless of data volume
Rule of thumb for interviews

"If a Salesforce admin should be able to understand and modify the logic without a code deployment, and it doesn't involve bulk operations or callouts, I use Flow. For everything that touches large data volumes, external APIs, or complex Apex logic — I use a trigger with handler pattern. The two approaches are complementary, not competing."

How do you create an @InvocableMethod for Flow — complete code with input and output variables?FLOW▼
// Complete InvocableMethod — Flow calls this as an Apex Action
public class LoanStatusNotificationAction {

    public class NotifyRequest {
        @InvocableVariable(label='Loan ID' required=true
            description='Salesforce record ID of the Loan__c to notify about')
        public Id loanId;

        @InvocableVariable(label='New Status' required=true)
        public String newStatus;

        @InvocableVariable(label='Send Email' description='Set to true to send borrower email')
        public Boolean sendEmail;
    }

    public class NotifyResult {
        @InvocableVariable(label='Notification Sent')
        public Boolean notificationSent;

        @InvocableVariable(label='Error Message')
        public String errorMessage;
    }

    // MUST accept and return List<T> — Flow can call in bulk
    @InvocableMethod(
        label='Send Loan Status Notification'
        description='Sends email/SMS to borrower when loan status changes. Use in Record-Triggered Flow after save.'
        category='Loan Processing'
    )
    public static List<NotifyResult> sendNotification(List<NotifyRequest> requests) {
        List<NotifyResult> results = new List<NotifyResult>();
        for(NotifyRequest req : requests) {
            NotifyResult result = new NotifyResult();
            try {
                if(req.sendEmail == true) {
                    NotificationService.sendStatusEmail(req.loanId, req.newStatus);
                }
                result.notificationSent = true;
            } catch(Exception e) {
                result.notificationSent = false;
                result.errorMessage = e.getMessage();
            }
            results.add(result);
        }
        return results;
    }
}

How do you debug a Flow that fails in production — what tools and approach?FLOW▼
  • Flow Builder Debug Button: Click Debug in Flow Builder → enter input values manually → step through element by element — see variable values at each step, error message at failing element
  • Debug Logs: Setup → Debug → Apex Debug Logs → Add trace for the user. Set FLOW = FINEST. Filter log for FLOW_ELEMENT_ERROR to find exact failure point and variable values
  • Send Flow Error Emails: Setup → Process Automation Settings → "Send Flow Error Emails" — admins receive an email with which flow, which element, the error message, and variable values at time of failure. HIGHLY recommended for production
  • Fault Paths: Every element that can fail (Get Records, Send Email, Apex Action) should have a Fault path. In the fault path: assign {!$Flow.FaultMessage} to a text variable, create an Error_Log__c record. Without fault paths, Flow fails silently
  • Rollback Mode: Debug → Run with Rollback Mode — tests the flow logic without actually committing DML. Safe for testing on real records
Production debugging tip

Never debug a failing Flow directly on production data. Reproduce the exact record state in Full Sandbox, debug there, fix, then deploy. If you must debug in production, use Rollback Mode to avoid unintended data changes.

What are the Flow governor limit pitfalls — when does Flow become dangerous at scale?FLOW▼
  • Get Records inside a Loop: Every iteration of a Loop with a Get Records element = one SOQL query. 100 loop iterations = 100 queries = governor limit hit. Solution: Get all records BEFORE the loop into a collection variable
  • Update Records inside a Loop: Every iteration = one DML. 150 iterations = limit hit. Solution: Add records to a collection variable inside the loop, Update Records AFTER the loop with the collection
  • After Save Flow on a frequently-updated object: If 1000 records are updated via Bulk API, the After Save Flow fires 1000 times. Each run shares the org's transaction limits. Solution: use "Only when a record is created or when a record meets/stops meeting filter conditions" to limit trigger firing
  • Recursive Flow: Flow updates a field on the record, which re-triggers the Record-Triggered Flow. Use entry conditions carefully. Use "Only when a record is created or meets criteria for the first time" (entry type) to prevent reentry
// BAD pattern in Flow loop — SOQL per iteration:
// Loop: For each Loan in Loan_Collection
//   Get Records: Get Contact WHERE Id = {!Current_Loan.BorrowerContact__c}  ← SOQL per iteration!

// GOOD pattern:
// Before loop: Get Records: Get all Contacts WHERE Id IN {!Loan_BorrowerContact_IDs}
// Loop: For each Loan in Loan_Collection
//   Decision: Find matching Contact in the collection (no SOQL inside loop)

What are Flow best practices for governor limit management?FLOW SCENARIOS▼
  • Before-Save flows have no governor limit impact — they run in same transaction as the save, no extra DML. Prefer Before-Save for simple field updates.
  • After-Save flows get 2,000 DML rows — but still count against the transaction's DML statement limit (150). Multiple After-Save flows on the same object share the limit.
  • Avoid loops that do DML inside — same rule as Apex. Use Collection variables to collect records, then a single Create/Update Records element outside the loop.
  • Scheduled Paths (async) — After-Save flows can have async paths that run after a delay. These get a fresh governor limit set — good for time-delayed actions.
  • Flow + Apex hybrid — complex logic goes in Apex @InvocableMethod, Flow handles the user interaction and orchestration. Each keeps to what it does best.

Collection performance

Use Record Collection Variables to batch DML in flows. Loop over records → add to collection variable → after loop use one Create/Update Records element on the whole collection. Never use DML elements inside loops.

How do sub-flows work — when should you extract logic into a sub-flow?FLOW SCENARIOS▼

Sub-flows are reusable flow components. Call them from a parent flow the same way you call a method — pass inputs, get outputs back.

// When to extract to sub-flow:
// 1. Same logic reused in 3+ flows
// 2. A logical unit that can be tested independently
// 3. Complex branching that makes the parent hard to read

// Sub-flow pattern:
// Parent flow: Loan Application Flow
//   → Sub-flow: Credit Check Sub-Flow (inputs: SSN, loanAmount. outputs: score, approved)
//   → Sub-flow: OFAC Check Sub-Flow (inputs: borrowerName, SSN. outputs: clear, matchFound)
//   → Sub-flow: Document Generation Sub-Flow (inputs: loanId, loanType)

// Benefits: each sub-flow can be updated independently without touching the parent
// Risk: deeply nested sub-flows are hard to debug — limit to 2 levels deep
Sub-flow governor limits

Sub-flows run in the SAME transaction as the parent. They share the same governor limit pool — a sub-flow that does 50 DML statements counts against the parent's 150-statement limit.

What is the Flow Orchestration — how does it differ from regular Flows?FLOW SCENARIOS▼

Flow Orchestration (formerly Orchestrator) enables multi-step, multi-user approval and review processes with assigned steps, stages, and work items.

  • Stages — phases of the orchestration (Application Review, Underwriting, Approval, Funding)
  • Steps — individual tasks within a stage. Can be assigned to specific users, roles, or queues
  • Interactive Steps — require a human to complete a screen flow (review and approve a document)
  • Background Steps — run automation automatically (call Apex, run a sub-flow) without human interaction
  • Sequential vs Parallel — steps in the same stage can run in sequence or in parallel
  • Work Items — tasks created for assigned users. Appear in the Salesforce To-Do list and Inbox
vs Standard Approval Process

Use Flow Orchestration when: multiple sequential approvers with different tasks, mixed human review and automated steps, need to track progress through stages, need complex routing logic. Use Standard Approval Process when: simple 1-3 level approval chain, no custom steps between approvals.
