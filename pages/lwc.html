What is the difference between @api, @track, and @wire — explain with code LWC
DecoratorPurposeAccessReactive?
@apiPublic property — accessible from parent component or Lightning App BuilderExternal — parent can set itYes
@trackReactive private property — triggers re-render when value changesInternal onlyYes (deep object/array)
@wireDeclarative data fetching — connects to wire adapters or cacheable ApexInternal, auto-populatedYes (auto-refires)
// Import decorators from lwc
import { LightningElement, api, track, wire } from 'lwc';
import getLoans from '@salesforce/apex/LoanController.getLoans';
import { getRecord, getFieldValue } from 'lightning/uiRecordApi';
import LOAN_AMOUNT_FIELD from '@salesforce/schema/Loan__c.LoanAmount__c';

export default class LoanDetails extends LightningElement {
    
    // @api - parent passes this in, re-renders when parent changes it
    @api recordId;          // Required for record pages
    @api loanType = 'FHA'; // With default value
    
    // @track - for deeply nested object changes
    // Note: in current LWC, plain properties ARE reactive for reassignment
    // @track only needed when mutating nested object/array properties
    @track filterCriteria = { minAmount: 0, maxAmount: 1000000, status: 'Active' };
    
    isLoading = false; // Plain property - reactive for reassignment (no @track needed)
    
    // @wire with Apex - requires cacheable=true on Apex method
    @wire(getLoans, { loanType: '$loanType' }) // '$loanType' = reactive param
    wiredLoans({ error, data }) {
        if(data) {
            this.loans = data;
            this.error = undefined;
        } else if(error) {
            this.error = error;
            this.loans = undefined;
        }
    }
    
    // @wire with standard adapter
    @wire(getRecord, { recordId: '$recordId', fields: [LOAN_AMOUNT_FIELD] })
    loanRecord;
    
    get loanAmount() {
        return getFieldValue(this.loanRecord.data, LOAN_AMOUNT_FIELD);
    }
}
Key distinction interviewers test

@track is no longer needed for primitive reassignment in modern LWC — all properties are reactive by default for assignment. @track is only needed when you mutate a nested property like this.obj.nestedProp = 'value' without reassigning the top-level object.

How do you communicate between LWC components — all 3 patterns with code LWC

Pattern 1: Parent ΓåÆ Child (via @api property or method call)

// Parent HTML - sets child @api property
<c-child-component loan-type={selectedType} loan-id={currentLoanId}></c-child-component>
<lightning-button label="Reset Child" onclick={resetChild}></lightning-button>

// Parent JS - call child method using querySelector
resetChild() {
    this.template.querySelector('c-child-component').resetForm(); // calls child's @api method
}

// Child JS - receives @api property
@api loanType;
@api resetForm() { // callable by parent
    this.formData = {};
    this.errors = [];
}

Pattern 2: Child ΓåÆ Parent (via CustomEvent)

// Child JS - dispatch event with data
handleSave() {
    const event = new CustomEvent('loansaved', {
        detail: { loanId: this.loanId, status: 'Approved' },
        bubbles: true,   // propagates up through DOM
        composed: false  // stays within component tree (false = default)
    });
    this.dispatchEvent(event);
}

// Parent HTML - listen for the event
<c-loan-form onloansaved={handleLoanSaved}></c-loan-form>

// Parent JS - handle event
handleLoanSaved(event) {
    const { loanId, status } = event.detail;
    this.showToast('Success', 'Loan ' + loanId + ' saved', 'success');
}

// Step 3: Subscriber component
import { subscribe, unsubscribe, APPLICATION_SCOPE, MessageContext } from 'lightning/messageService';
import LOAN_STATUS_CHANNEL from '@salesforce/messageChannel/LoanStatusChannel__c';

subscription = null;

connectedCallback() {
    this.subscription = subscribe(this.messageContext, LOAN_STATUS_CHANNEL, 
        (msg) => this.handleLoanUpdate(msg), { scope: APPLICATION_SCOPE });
}
disconnectedCallback() {
    unsubscribe(this.subscription); // Always unsubscribe to prevent memory leaks
}
handleLoanUpdate(message) {
    this.currentLoanId = message.loanId;
    this.loanStatus = message.status;
}
LWC lifecycle hooks — what fires in what order? LWC

Order of lifecycle hooks: constructor ΓåÆ connectedCallback ΓåÆ render ΓåÆ renderedCallback

export default class LoanComponent extends LightningElement {
    
    constructor() {
        super(); // ALWAYS call super() first - mandatory
        // Fires: When component is instantiated
        // Cannot: Access DOM (not rendered yet), access @api properties
        // Can: Set private properties, initialize collections
        this.loans = [];
    }
    
    connectedCallback() {
        // Fires: When component is inserted into the DOM
        // Can: Access template (careful - may not be fully rendered)
        // Use for: Initialization logic, subscriptions, imperative Apex calls
        this.loadInitialData();
        this.registerListeners(); // LMS subscriptions go here
    }
    
    renderedCallback() {
        // Fires: After every render (initial and re-renders)
        // WARNING: Can cause infinite loop if you modify state here without a guard
        // Use hasRendered flag to run once:
        if(this.hasRendered) return;
        this.hasRendered = true;
        // Third-party JS initialization (charts, maps) goes here - DOM is ready
        this.initializeChart();
    }
    
    disconnectedCallback() {
        // Fires: When component is removed from DOM
        // Use for: Cleanup - unsubscribe LMS, clear intervals/timeouts
        unsubscribe(this.subscription);
        clearInterval(this.pollingInterval);
    }
    
    errorCallback(error, stack) {
        // Fires: When child component throws an error
        // Use for: Graceful error handling, error boundary pattern
        this.errorMessage = error.message;
    }
}

// WIRE - declarative, auto-fires, cached
@wire(getLoansByStatus, { status: '$selectedStatus' }) // fires when selectedStatus changes
wiredLoans({ data, error }) {
    this.isLoading = false;
    if(data) { this.loans = data; this.error = undefined; }
    else if(error) { this.error = error.body.message; }
}

// Refresh wire data after DML:
import { refreshApex } from '@salesforce/apex';
handleSave() {
    saveLoan({ loanData: this.loanRecord })
        .then(() => refreshApex(this.wiredLoans)) // force wire to refetch
        .catch(error => this.handleError(error));
}

// IMPERATIVE - user-triggered, no caching, more control
handleSearch() {
    this.isLoading = true;
    searchLoans({ searchTerm: this.searchTerm, minAmount: this.minAmount })
        .then(result => {
            this.loans = result;
            this.isLoading = false;
        })
        .catch(error => {
            this.error = error.body.message;
            this.isLoading = false;
        });
}

// Or with async/await:
async handleSaveLoan() {
    try {
        this.isLoading = true;
        const result = await saveLoan({ loan: this.loanData });
        this.showToast('Success', 'Loan saved: ' + result.Id, 'success');
        await refreshApex(this.wiredLoans);
    } catch(error) {
        this.showToast('Error', error.body.message, 'error');
    } finally {
        this.isLoading = false;
    }
}
