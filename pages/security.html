Explain the Salesforce security model layers from top to bottom SECURITY

Salesforce security is layered — each layer controls a different level of access:

  • Organization Level: Login hours, IP restrictions, password policies, 2FA settings
  • Object Level (CRUD): Profiles and Permission Sets control Create, Read, Edit, Delete per object
  • Field Level (FLS): Profiles and Permission Sets control visibility and editability of individual fields
  • Record Level (Sharing): OWD, Role Hierarchy, Sharing Rules, Manual Sharing, Apex Sharing control which records a user can see

Important rule: These layers work together — a user must pass ALL layers. Even if sharing rules give record access, FLS still controls which fields are visible. Even if CRUD gives object access, OWD may restrict which records.

How to remember it

Think of it as security gates: Org gate ΓåÆ Object gate ΓåÆ Field gate ΓåÆ Record gate. A user must pass through all gates to see a specific field value on a specific record.

What is OWD? Explain each setting and when you use it. SECURITY

OWD (Organization-Wide Default) is the baseline record access level for every object. It sets the most restrictive access level, and you then open up access using Role Hierarchy, Sharing Rules, or Manual Sharing.

OWD SettingWhat It MeansUse Case
PrivateOnly record owner (and users above in role hierarchy) can see/editLoan applications — only assigned loan officer should see their loans
Public Read OnlyEveryone can read, only owner can editRate sheets, product catalog that all can see but only admins edit
Public Read/WriteEveryone can read and editShared lookup tables, common reference data
Controlled by ParentInherits access from parent objectContact inheriting Account's sharing — if you can see Account, you can see its Contacts
Key principle

Always set OWD to the most restrictive setting that makes sense for the object, then use sharing rules and role hierarchy to open up access. Never start with Public and try to restrict down — that's architecturally wrong and leads to data exposure.

What is with sharing, without sharing, inherited sharing — explain with code SECURITY
KeywordBehaviorWhen to Use
with sharingEnforces the running user's sharing rules in SOQL/DML — user only sees records they have access toDefault for all service classes — always use this unless there's a reason not to
without sharingRuns in system mode — ignores sharing rules, user sees all recordsBatch jobs, system processes, when you explicitly need access beyond user's permissions
inherited sharingInherits sharing context from the calling classUtility/helper classes that should adapt to their caller's sharing context
(no keyword)Defaults to without sharing for classes called from system context (triggers, batch)Avoid — be explicit about sharing intent
// Service class - always explicitly declare sharing
public with sharing class LoanService {
    // Enforces user's sharing - borrowers can't see other borrowers' loans
    public static List<Loan__c> getUserLoans() {
        return [SELECT Id, LoanAmount__c, Status__c FROM Loan__c];
        // Only returns loans this user can see based on OWD + sharing rules
    }
}

// Batch job - needs system-level access to process all records
public without sharing class LoanInterestBatch implements Database.Batchable<SObject> {
    // Processes ALL loans regardless of ownership - needed for system batch
}

// Utility class - adapts to caller's sharing
public inherited sharing class ValidationUtils {
    // If called from with sharing context, respects sharing
    // If called from without sharing context, runs in system mode
    public static Boolean isValidLoanAmount(Decimal amount) {
        return amount > 0 && amount <= 5000000;
    }
}
How do you enforce FLS in Apex? What is stripInaccessible? SECURITY 
// Method 1: Manual FLS check before DML
public static void updateLoanStatus(Loan__c loan) {
    Schema.SObjectField statusField = Loan__c.SObjectType.getDescribe()
                                       .fields.getMap().get('Status__c');
    if(!statusField.getDescribe().isUpdateable()) {
        throw new SecurityException('You do not have permission to update Status field');
    }
    update loan;
}

// Method 2: stripInaccessible - BEST for bulk operations
// Automatically removes fields the user cannot access - no exception thrown
public static void safeInsertContacts(List<Contact> contacts) {
    // Before DML - strips fields user cannot edit
    SObjectAccessDecision decision = Security.stripInaccessible(
        AccessType.CREATABLE,  // CREATABLE, UPDATABLE, READABLE, UPSERTABLE
        contacts
    );
    insert decision.getRecords(); // only accessible fields are set
}

// Method 3: WITH USER_MODE in SOQL (clean, modern)
List<Loan__c> loans = [
    SELECT Id, LoanAmount__c, SSN__c 
    FROM Loan__c 
    WITH USER_MODE  // silently excludes fields user cannot see
];

// CRUD check before DML
if(!Loan__c.sObjectType.getDescribe().isCreateable()) {
    throw new SecurityException('No permission to create Loan records');
}
When to use stripInaccessible

Use stripInaccessible when displaying or inserting records from user input — like when saving a form submitted by a Community user. It silently cleans the data rather than throwing errors, making for a better user experience while maintaining security.
