Advanced Async — Experian Credit Pull Architecture
Complex real-world implementation involving Queueable, Platform Events, and External APIs
Design a high-reliability credit pull system for a mortgage company. How do you handle failures?ASYNC

The Business Problem: Experian API takes 15-45 seconds to return a full mortgage credit report. You can't do this in a standard trigger or LWC wire call without blocking the UI or hitting CPU limits.

The Solution: Decoupled Event-Driven Architecture.

  1. Trigger Phase: When Loan Status → 'Needs Credit', publish Credit_Pull_Request__e Platform Event.
  2. Async Phase: Platform Event Trigger catches event, enqueues ExperianQueueable.
  3. API Phase: Queueable makes callout, receives JSON.
  4. Storage Phase: If success, parse and update Loan record. If timeout, write to Sync_Retry_Queue__c.
public class ExperianQueueable implements Queueable, Database.AllowsCallouts {
    private Id loanId;
    public ExperianQueueable(Id lId) { this.loanId = lId; }

    public void execute(QueueableContext ctx) {
        try {
            // 1. Call API
            HttpResponse res = ExperianAPI.pull(loanId);
            
            // 2. Handle Success
            if (res.getStatusCode() == 200) {
                ExperianParser.updateLoan(loanId, res.getBody());
            } else {
                // 3. Handle Business Error (Retry-able)
                SyncService.retryLater(loanId, 'Experian', res.getStatus());
            }
        } catch (Exception e) {
            // 4. Handle Critical Error
            ErrorLog.log('Experian Failure', e.getMessage(), loanId);
        }
    }
}
Why use Platform Events to trigger the Queueable? Why not call Queueable directly from the trigger?ASYNC

Calling Queueable directly from a trigger is okay for low volumes, but Platform Events offer Transaction Isolation and Governor Limit Resets.

  • Isolation: If the main loan save fails and rolls back, the Platform Event (if set to "Publish After Commit") won't fire. We don't pull credit for a loan that wasn't actually saved.
  • Limits: Queueable limit is 50 per transaction. Platform Events run in their own context, allowing much higher scale without hitting the limit of the main transaction.
  • Decoupling: The loan trigger doesn't need to know how credit is pulled. It just says "hey, credit is needed." Multiple subscribers can react independently.
How do you handle "Read Timeout" or "API Unavailable" errors in a production batch sync?ASYNC

Exponential Backoff Retry Pattern.

  1. Store failed attempts in a Integration_Log__c record related to the Loan.
  2. Increment a Retry_Count__c field.
  3. A Scheduled Apex job runs every 15 minutes, finds records where Retry_Count__c < 3 and Next_Retry__c <= NOW().
  4. Backoff Logic: Retry 1 (15m), Retry 2 (1h), Retry 3 (4h).
// Scheduled job logic
public void execute(SchedulableContext sc) {
    List<Loan__c> toRetry = [SELECT Id FROM Loan__c WHERE Needs_Retry__c = true LIMIT 50];
    for (Loan__c l : toRetry) {
        System.enqueueJob(new ExperianQueueable(l.Id));
    }
}

Senior Architect Rule

If you're processing a list of things from a query → Batch. If you're reacting to a single user action or trigger event → Queueable.
