Official Guideline

Salesforce's current stance is "Flow First." Try to build it in Flow. If you hit a limit, or the logic becomes a "spaghetti" of elements that is hard to maintain, move that specific part to an Invocable Apex action or rewrite the whole process in Apex.

Record-Triggered Flow: Before-Save vs After-Save — explain the difference?FLOW▼
  • Before-Save (Fast Field Updates): Runs before the record is saved to the database. Use this for updating fields on the SAME record that triggered the flow. It is 10x faster than After-Save because it doesn't trigger a second save operation.
  • After-Save (Actions and Related Records): Runs after the record is saved. Use this for updating OTHER records, sending emails, making callouts, or performing any action that needs the record's ID (which is only available after save).

How do you handle Errors in Flow — use Fault Paths and custom error messages?FLOW▼

Never let a Flow fail with a "unhandled exception" screen. Use Fault Paths.

  • Fault Path: Connect the 'Fault' output of a DML element (Create, Update, Delete) to a separate logic branch.
  • Custom Error Element (2024): Use the "Custom Error" element in the fault path to show a user-friendly error message directly on the screen or field, preventing the save.
  • Logging: In the fault path, call an Apex Action to log the error to a custom "Error_Log__c" object so admins can debug later.
  • Email Alert: Send a notification to the admin team with the {!$Flow.FaultMessage} global variable.

What is Flow Orchestration — when do you use it instead of a regular Flow?FLOW▼

Orchestrator handles complex, multi-user, multi-step processes that take days or weeks to complete. A regular Flow is typically "one user, one session."

  • Stages and Steps: Orchestration is broken into Stages (e.g., "Application Review," "Underwriting," "Closing"). Each stage contains multiple Steps.
  • Interactive Steps: Assign a task to a specific user (e.g., "Underwriter John needs to approve this document"). The process pauses until John completes his Screen Flow.
  • Background Steps: Run automated logic between human steps (e.g., "After approval, send document to DocuSign").
  • Monitoring: Admins can see exactly where a loan is "stuck" in the orchestration timeline.
How do you handle "Loops" in Flow without hitting governor limits?FLOW▼
  • Bulkification: NEVER put a "Create Records" or "Update Records" element INSIDE a loop.
  • Correct Pattern:
    1. Loop through records.
    2. Inside loop: use an 'Assignment' element to add values to a Record Collection Variable.
    3. After the loop: use a single "Update Records" element on that collection.
  • Limits: A single Flow transaction can execute a maximum of 2,000 elements. If your loop processes 500 records and has 5 elements inside, you will hit the 2,500 element limit (500 * 5) and the flow will fail. In this case, move the logic to Apex.
How do you migrate Workflow Rules and Process Builders to Flow — "Migrate to Flow" tool?FLOW▼
  • The Tool: Setup → Migrate to Flow. Select your old Workflow Rule or Process Builder and the tool generates a Flow automatically.
  • Deactivation: After migration, you MUST deactivate the old rule/process and activate the new Flow. Never have both running at once.
  • Consolidation: Don't just migrate 1:1. Use the migration as an opportunity to consolidate multiple processes into one "Trigger Handler" Flow for that object to improve performance and maintainability.
  • Limitations: The tool doesn't handle everything (e.g., complex formulas or certain actions). Always test the generated Flow thoroughly in a sandbox.
