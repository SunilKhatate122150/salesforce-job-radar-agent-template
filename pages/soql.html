What is SOQL injection and how do you prevent it? Show vulnerable code and the fix. SOQL

SOQL injection is a security vulnerability where a malicious user inputs text that manipulates a dynamic SOQL query to expose unauthorized data — similar to SQL injection in traditional databases.

// VULNERABLE CODE - Never do this
public static List<Account> searchAccounts(String name) {
    String query = 'SELECT Id, Name FROM Account WHERE Name = \'' + name + '\'';
    return Database.query(query);
    // If name = "test' OR Name != '", this returns ALL accounts!
}

// FIX 1: escapeSingleQuotes() - escapes the user input
public static List<Account> searchAccounts(String name) {
    String safeName = String.escapeSingleQuotes(name);
    String query = 'SELECT Id, Name FROM Account WHERE Name = \'' + safeName + '\'';
    return Database.query(query);
}

// FIX 2: Bind variables in dynamic SOQL (best approach)
public static List<Account> searchAccounts(String name) {
    // :name is a bind variable - Salesforce handles escaping automatically
    return Database.query('SELECT Id, Name FROM Account WHERE Name = :name');
}

// FIX 3: Use static SOQL instead of dynamic when possible
return [SELECT Id, Name FROM Account WHERE Name = :name];
Best practice order

1. Use static SOQL (not dynamic) whenever possible. 2. If dynamic SOQL is needed, use bind variable :name syntax. 3. If you must concatenate, always use escapeSingleQuotes(). 4. Validate and whitelist the input (if filtering by Status__c, verify the value is in an allowed list).

Write a SOQL query to get all Contacts with their Account name and parent Account. Explain relationship queries. SOQL

Salesforce has two types of relationship queries:

1. Child-to-Parent (traversing up): Use dot notation. You traverse from the child up to the parent. The relationship name depends on the field.

// Child-to-Parent: Contact ΓåÆ Account ΓåÆ Account's Parent
List<Contact> contacts = [
    SELECT Id, 
           FirstName, 
           LastName,
           Account.Name,                  // parent Account name
           Account.Phone,                 // parent Account phone
           Account.Parent.Name,           // grandparent Account name
           Account.Parent.Industry        // grandparent field (up to 5 levels)
    FROM Contact
    WHERE Account.Type = 'Broker'
];

// Accessing in Apex:
for(Contact c : contacts) {
    System.debug(c.Account.Name);          // parent account name
    System.debug(c.Account.Parent.Name);   // grandparent name
}

2. Parent-to-Child (subquery - traversing down): Use relationship name in a subquery. For standard objects the plural of the child object name. For custom, use __r (relationship) name.

// Parent-to-Child: Account with all its Contacts (subquery)
List<Account> accounts = [
    SELECT Id, Name, 
           (SELECT Id, FirstName, LastName, Email FROM Contacts),       // standard child
           (SELECT Id, CloseDate, Amount FROM Opportunities WHERE StageName != 'Closed Lost'),
           (SELECT Id, LoanAmount__c FROM Loans__r)                     // custom child (__r)
    FROM Account
    WHERE Id IN :accountIds
];

// Accessing child records:
for(Account a : accounts) {
    for(Contact c : a.Contacts) {  // iterating over subquery results
        System.debug(c.Email);
    }
}
Explain aggregate functions in SOQL — GROUP BY, HAVING, AggregateResult SOQL 
// COUNT, SUM, MIN, MAX, AVG - aggregate functions
List<AggregateResult> results = [
    SELECT Account.Name accName, 
           COUNT(Id) loanCount,
           SUM(LoanAmount__c) totalAmount,
           AVG(LoanAmount__c) avgAmount,
           MAX(LoanAmount__c) maxLoan
    FROM Loan__c
    WHERE Status__c != 'Denied'
    GROUP BY Account.Name
    HAVING COUNT(Id) > 5          // HAVING filters on aggregate result (like WHERE for GROUP BY)
    ORDER BY SUM(LoanAmount__c) DESC
    LIMIT 20
];

// Accessing AggregateResult - use alias or field name as string
for(AggregateResult ar : results) {
    String name = (String) ar.get('accName');
    Integer count = (Integer) ar.get('loanCount');
    Decimal total = (Decimal) ar.get('totalAmount');
    System.debug(name + ': ' + count + ' loans, total: ' + total);
}

// COUNT() vs COUNT(Id)
// COUNT() - returns Integer, no GROUP BY needed, no field needed
Integer total = [SELECT COUNT() FROM Account WHERE Industry = 'Financial'];
// COUNT(Id) - returns AggregateResult, can be used with GROUP BY
HAVING vs WHERE

WHERE filters individual records BEFORE grouping. HAVING filters the grouped results AFTER aggregation. You cannot use WHERE on an aggregate function — use HAVING for that.

What is WITH SECURITY_ENFORCED and WITH USER_MODE — what is the difference? SOQL

Both enforce field-level security in SOQL queries, but they work differently:

FeatureWITH SECURITY_ENFORCEDWITH USER_MODE
Available sinceAPI 48.0 (Spring '20)API 56.0 (Spring '23)
If inaccessible field queriedThrows QueryException — all or nothingStrips inaccessible fields silently, no error
DML enforcementNoYes — also enforces CRUD on DML
UsageIn SOQL query clauseAfter Database.query() or as class modifier
// WITH SECURITY_ENFORCED - throws QueryException if any field is not accessible
List<Account> accs = [SELECT Id, Name, AnnualRevenue 
                       FROM Account WITH SECURITY_ENFORCED];
// Limitation: does NOT work in subqueries, does NOT work with GROUP BY

// WITH USER_MODE - more flexible, newer approach
List<Account> accs = [SELECT Id, Name, AnnualRevenue 
                       FROM Account WITH USER_MODE];

// WITH USER_MODE can also be applied to an entire class
public with sharing class AccountService {
    public static List<Account> getAccounts() {
        return Database.query('SELECT Id, Name FROM Account', AccessLevel.USER_MODE);
    }
}
Recommendation

Prefer WITH USER_MODE for new code. It's more flexible, supports subqueries, and also enforces CRUD during DML. WITH SECURITY_ENFORCED is older and has limitations with subqueries and aggregate functions.

What is SOQL FOR UPDATE — row locking? When do you use it? SOQL

FOR UPDATE locks the queried records in the database, preventing other transactions from modifying them until your current transaction completes. It is pessimistic locking — assume conflict and prevent it.

// FOR UPDATE - locks records while you process them
List<Loan__c> loans = [SELECT Id, Status__c, LoanAmount__c 
                        FROM Loan__c 
                        WHERE Status__c = 'Underwriting'
                        FOR UPDATE]; // Locked! No one else can modify these

// Process and update - other transactions will wait or fail
for(Loan__c l : loans) {
    l.Status__c = 'Approved';
    l.ApprovedDate__c = Date.today();
}
update loans; // Lock released after DML completes

When to use FOR UPDATE:

  • Financial transactions where double-processing must be prevented
  • Status transitions that should only happen once (e.g., loan approval)
  • Counter/accumulator fields updated by multiple concurrent users
Deadlock risk

If two transactions lock different sets of records and then try to access each other's locked records, a deadlock occurs. Salesforce will throw a QueryException. Always lock records in a consistent order and keep locked transactions short.
