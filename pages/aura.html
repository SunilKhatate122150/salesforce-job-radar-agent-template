Interview answer tip

Say: "LWC is based on web standards, which means faster performance and easier debugging with browser dev tools. Aura is a proprietary framework that predates web components. In current projects, I always build in LWC and only touch Aura when maintaining legacy components. Salesforce itself is investing all new features in LWC."

How do you call Apex from an Aura component — complete callback pattern?AURA

All Aura Apex calls are asynchronous and use a callback pattern (unlike LWC async/await).

// MyComp.cmp
<aura:component controller="LoanController">
  <aura:attribute name="loans" type="Loan__c[]"/>
  <aura:attribute name="isLoading" type="Boolean" default="false"/>
  <aura:handler name="init" value="{!this}" action="{!c.doInit}"/>
  <aura:iteration items="{!v.loans}" var="loan">
    <p>{!loan.Name}</p>
  </aura:iteration>
</aura:component>

// MyCompController.js
({
  doInit: function(component, event, helper) {
    component.set('v.isLoading', true);
    var action = component.get('c.getLoans');  // c. = Apex controller
    action.setParams({ status: 'Active' });
    action.setCallback(this, function(response) {
      component.set('v.isLoading', false);
      if (response.getState() === 'SUCCESS') {
        component.set('v.loans', response.getReturnValue());
      } else {
        var errors = response.getError();
        console.error('Error:', errors[0].message);
      }
    });
    action.setStorable(); // cache result (like cacheable=true in LWC)
    $A.enqueueAction(action);
  }
})
What are Aura component events vs application events — when to use each?AURA
TypeScopeDirectionUse when
Component EventBubbles up through component hierarchyChild ΓåÆ Parent ancestor onlyParent needs to react to child action
Application EventBroadcast to ALL handlers in entire appAny component ΓåÆ any other componentTruly unrelated components need to communicate
// Component Event (preferred — more contained)
// LoanSelectedEvt.evt: <aura:event type="COMPONENT"><aura:attribute name="loanId" type="String"/></aura:event>
// Fire: component.getEvent('loanSelected').setParams({loanId: id}).fire();
// Handle in parent: <aura:handler name="loanSelected" event="c:LoanSelectedEvt" action="{!c.onSelect}"/>

// Application Event (use sparingly)
// <aura:event type="APPLICATION"><aura:attribute name="loanId" type="String"/></aura:event>
// Fire: $A.get('e.c:LoanApprovedEvt').setParams({loanId: id}).fire();
// Handle anywhere: <aura:handler event="c:LoanApprovedEvt" action="{!c.onApproval}"/>
Application events create tight coupling

They fire to every registered handler in the app. In large orgs this makes debugging very hard. Prefer Component Events or migrate to LWC + Lightning Message Service.

What is Aura Locker Service — what restrictions does it impose?AURA

Locker Service is Salesforce's JavaScript security sandbox for Aura. It wraps DOM elements in SecureElement proxies to prevent cross-component DOM access and XSS.

  • Cannot use document.querySelector() — blocked. Use component.find('aura:id') instead
  • instanceof checks fail — element instanceof HTMLInputElement returns false (it's a SecureElement). Check tagName instead
  • eval() and Function() blocked — cannot dynamically execute code strings
  • Cross-namespace DOM access blocked — cannot access another component's internal DOM
  • Some browser globals restricted — window, document proxied through Locker
// WRONG — blocked by Locker
var input = document.getElementById('myInput');
if (input instanceof HTMLInputElement) { } // always false

// CORRECT — Aura safe patterns
var input = component.find('myInput'); // find by aura:id
if (input.getElement().tagName === 'INPUT') { } // check tagName not instanceof
How do Aura components communicate with Lightning Web Components?AURA

LWC components can be used inside Aura components but the reverse is not true — you cannot use Aura inside LWC.

// Using LWC inside Aura — works natively
<aura:component>
  <!-- c-loan-card is an LWC component -->
  <c-loan-card loan-id="{!v.selectedLoanId}" onloanapproved="{!c.handleApproval}"></c-loan-card>
</aura:component>

// LWC child communicates to Aura parent via Custom Events — Aura listens with on prefix
// LWC fires: this.dispatchEvent(new CustomEvent('loanapproved', { detail: { loanId } }));
// Aura handles: onloanapproved="{!c.handleApproval}"
// handleApproval: function(cmp, event) { var loanId = event.getParam('detail').loanId; }

// Lightning Message Service (LMS) — works for both Aura and LWC
// Best for two-way communication between Aura and LWC components on the same page
Migration strategy

Wrap existing Aura components in LWC wrappers gradually. New features = LWC. Existing Aura = maintain until full replacement. Use LMS as the communication bridge during transition.

How do you use $A.createComponent() for dynamic component creation?AURA
// Dynamically create a component at runtime (no LWC equivalent — Aura-only)
({
  createLoanModal: function(component, event, helper) {
    $A.createComponent(
      'c:LoanApprovalModal',  // component type
      {                        // attribute values
        'loanId': component.get('v.selectedLoanId'),
        'onClose': component.getReference('c.handleModalClose')
      },
      function(newCmp, status, errorMessage) {
        if (status === 'SUCCESS') {
          var body = component.get('v.body');
          body.push(newCmp);
          component.set('v.body', body); // add to component body
        } else {
          console.error('Component creation failed:', errorMessage);
        }
      }
    );
  }
})
When used

$A.createComponent is used for modal dialogs, dynamic content areas, and wizard steps where the component type is determined at runtime. In LWC, use conditional templates (if:true) or dynamic imports instead.

What is Aura helper — how does it differ from controller?AURA

Every Aura component has a Controller (event handlers, bound to markup) and a Helper (shared utility functions). The Helper is a singleton — one instance shared across all instances of the component.

// Controller — handles events from markup, delegates to helper
({
  handleSave: function(component, event, helper) {
    helper.saveRecord(component, function(result) {
      component.set('v.message', 'Saved: ' + result.Id);
    });
  }
})

// Helper — reusable utility methods, shared across all component instances
({
  saveRecord: function(component, callback) {
    var action = component.get('c.saveLoan');
    action.setParams({ loan: component.get('v.loan') });
    action.setCallback(this, function(response) {
      if (response.getState() === 'SUCCESS') callback(response.getReturnValue());
    });
    $A.enqueueAction(action);
  },
  validateForm: function(component) {
    var amount = component.get('v.loan').Amount__c;
    return amount > 0 && amount <= 10000000;
  }
})
Best practice

Keep Controller functions thin — they just call Helper methods. This makes Helper functions easily testable and reusable.

How do you handle errors in Aura — force:showToast and error events?AURA
// Show toast notification
({
  showSuccess: function(component) {
    var toastEvent = $A.get('e.force:showToast');
    toastEvent.setParams({
      title: 'Success',
      message: 'Loan approved successfully.',
      type: 'success',   // success, error, warning, info
      mode: 'dismissible' // dismissible, sticky, pester
    });
    toastEvent.fire();
  },

  handleError: function(component, error) {
    var toastEvent = $A.get('e.force:showToast');
    toastEvent.setParams({
      title: 'Error',
      message: error.message || 'An unexpected error occurred.',
      type: 'error',
      mode: 'sticky'
    });
    toastEvent.fire();
  },

  // Handle Apex error in callback
  handleApexResponse: function(component, response) {
    if (response.getState() === 'ERROR') {
      var errors = response.getError();
      var message = (errors && errors[0] && errors[0].message) ? errors[0].message : 'Unknown error';
      this.handleError(component, { message: message });
    }
  }
})
