Design a notification system in Salesforce that sends emails, SMS, and push notifications based on loan status changes - how would you architect this? SCENARIO

Step 1 - Identify the trigger: Record-Triggered Flow or After Update trigger on Loan__c fires when Status__c changes.

Step 2 - Decouple with Platform Events: Don't call notification services directly from the trigger. Instead, publish a Loan_Status_Changed__e Platform Event. This decouples the loan record update from the notification logic - if the notification service is down, the loan record still saves successfully.

Step 3 - Notification Router: A Platform Event trigger subscribes to Loan_Status_Changed__e and calls a NotificationRouter class. The router reads Notification_Config__mdt (custom metadata) to determine which channels to use for each loan type + status combination.

Step 4 - Channel adapters: Each channel (email, SMS via Twilio, push via Salesforce Notification Builder) is implemented as a separate class behind an INotificationChannel interface. The router loops through the configured channels and calls each.

Step 5 - Error handling: Use a Notification_Log__c custom object to record every notification attempt - status, timestamp, error if any. A scheduled job retries failed notifications up to 3 times.

Key patterns demonstrated

Platform Events (decoupling), Custom Metadata (config-driven, no code changes for new rules), Strategy pattern (INotificationChannel), Error logging, Retry logic. This answer shows senior architect thinking - not just "call Messaging.sendEmail."

A batch job that processes 2 million loan records is timing out. How do you diagnose and fix it? SCENARIO

Step 1 - Diagnose first: Check the AsyncApexJob record to see which chunk failed. Enable debug logs for the batch user. Look at which line hits the time limit - use Limits.getCpuTime() at different points to narrow it down.

Common causes for slow batch on 2M records:

  • SOQL inside execute() loop - query runs per chunk instead of once
  • Scope size too large (default 200) - each chunk takes too long. Try smaller scope: Database.executeBatch(job, 50)
  • Triggers firing on DML inside execute() - your batch update fires a trigger that does expensive work per record
  • Complex formula fields being evaluated on every query
  • String concatenation or regex in a loop inside execute()

Fixes:

  • Reduce scope size - more chunks, less work per chunk
  • Move static SOQL lookups to start() - pass data into execute() via instance variables and Database.Stateful
  • Use TriggerHandler.bypass() before the update DML in execute() to skip expensive trigger logic during batch runs
  • Split into multiple smaller batch jobs chained from finish()
  • Use Database.QueryLocator with a more selective WHERE clause - process only records that actually need changes
How would you migrate 500,000 records from a legacy system into Salesforce without affecting production users? SCENARIO

Strategy: Phased migration with validation gates

  • Phase 1 - Mapping: Map source system fields to Salesforce fields. Identify data quality issues in source (nulls, duplicates, invalid formats). Clean data before migrating.
  • Phase 2 - Sandbox test: Load 1,000 sample records into Full Sandbox. Verify triggers, workflows, validation rules behave correctly. Fix any issues found.
  • Phase 3 - Disable automation: Before production load, bypass triggers using a custom Boolean field or TriggerHandler.bypass(). Disable validation rules that would block historical data (e.g., TRID date fields from 2010 don't meet today's rules). Use a Migration Mode custom setting to conditionally skip logic.
  • Phase 4 - Load in batches: Use Salesforce Bulk API 2.0 - it handles 500K records efficiently. Load in batches of 10,000 with error logs. Do not load everything in one job - if it fails mid-way, you need to know exactly which records succeeded.
  • Phase 5 - Validate after load: Run COUNT queries to verify record counts. Spot-check 50 random records against source system. Run data quality reports.
  • Phase 6 - Re-enable automation: Turn triggers and validation rules back on. Monitor for the first week post-migration for unexpected issues.
How would you design bidirectional sync between Salesforce and a Loan Origination System?SCENARIO

Salesforce to LOS (outbound):

  • After-Save Record-Triggered Flow or trigger detects change on Loan__c
  • Publishes Loan_Sync_Request__e Platform Event - decouples loan save from LOS call
  • Platform Event subscriber trigger enqueues Queueable with Database.AllowsCallouts
  • Queueable makes REST callout to LOS API with idempotency key
  • On failure: writes to Sync_Retry_Queue__c. Scheduled Apex retries every 15 min up to 3 times

LOS to Salesforce (inbound):

  • LOS sends webhook: POST /services/apexrest/v1/loanSync
  • Apex @RestResource validates HMAC signature, upserts using LOS_ID__c External ID field
  • Returns HTTP 200 success, 400 bad payload, 500 error

Conflict resolution: Last_Sync_Source__c field (LOS or SFDC) + Last_Sync_Timestamp__c. Business rule: Salesforce wins for status changes, LOS wins for rate locks and financial figures.
