Tell me about the most complex project you've built PROJECT STORY
Situation
Our mortgage company needed loan officers to run targeted email campaigns to prospective borrowers, but they had no way to personalize content based on current market conditions like interest rates. The manual process was completely disconnected from real-time data.
Task
I was assigned as the sole developer to design and build this platform end-to-end — from architecture decisions to deployment. No other developer. I had to integrate multiple external APIs, build a full LWC UI, and ensure the system worked within Salesforce's governor limits at scale.
Action
I architected the platform using a service layer pattern to keep the integration logic completely separated from the Salesforce data layer. I integrated four external APIs: Resend for email delivery because it had better deliverability and a clean API compared to Salesforce's built-in email; OpenAI GPT-4o for AI-generated personalized email content based on borrower profile and loan type; the FRED API from the Federal Reserve to pull live economic indicators like unemployment and inflation data; and Freddie Mac's rate service for real-time mortgage rate feeds. I used Named Credentials for all authentication, Queueable Apex for async email processing to stay within callout limits, and built the entire front-end in LWC with a real-time rate dashboard and campaign builder interface. I wrote comprehensive test classes achieving 90% coverage.
Result
Impact
The platform launched on schedule. Loan officers could create and send personalized campaigns in 10 minutes compared to hours previously. Email open rates improved by around 35% due to AI-generated personalized content. I delivered this entirely solo — demonstrating ownership, architecture skills, and technical depth across the full Salesforce stack.

Impact
Credit pull time reduced from 25 minutes to under 30 seconds. Data entry errors were eliminated. The FCRA compliance check prevented unauthorized pulls, reducing compliance risk significantly. The integration also served as the template for our OFAC screening integration that followed.
Tell me about a time you debugged a critical production issue PROBLEM SOLVING
Situation
In production, loan officers started reporting that some loans were getting stuck in "Processing" status and not automatically advancing to "Underwriting" after all required documents were attached. This was happening intermittently — about 15-20% of qualifying loans were affected. This was critical because delayed processing meant delayed closings, which upset borrowers and could cause regulatory issues.
Task
I needed to identify root cause quickly — the business was escalating the issue. I had to diagnose a bug that was intermittent, in a trigger that ran on file attachment, in a production org I couldn't deploy to without change management approval.
Action
First, I checked the Salesforce Debug Logs for the affected loan IDs — I found several with a "governor limit exception" for SOQL queries but it was being swallowed by an empty catch block. The trigger was counting document types attached to the loan, but the query was inside a for loop — so when 6 or more documents were attached at once (which happened when users did bulk uploads), the 100 SOQL limit was hit and the exception was silently caught and ignored. I fixed the bug in sandbox by refactoring to collect all Loan IDs first, then use a single aggregate SOQL query to count documents per loan. I also removed the empty catch block and replaced it with proper error logging to a custom Error Log object. I created a quick emergency fix and worked with the team to get it deployed through a hotfix process. I also wrote a data fix script to reprocess the stuck loans.
Result
Impact
Bug was identified within 2 hours and fixed within 4 hours. 47 stuck loans were reprocessed. I added a code review rule to the team: never empty catch blocks, always test with 200 records in bulk. This incident became a learning session I led for the team on governor limits and bulkification.

Tell me about the most complex bug you debugged?BEHAVIORAL

Situation: In production, loan status intermittently did not update even though LWC showed Save Successful. Happened 3% of the time — no reproducible steps.

Action: I ruled out obvious causes — no failed DML, no validation failures, no sharing issues. The 3% rate correlated with concurrent edits — two users updating the same record within seconds. Found that our static variable recursion guard was set to true even when the first trigger invocation failed, and was never reset in a finally block. When User B saved immediately after, the guard was still true and the handler exited early.

Result: After resetting the flag in a finally block, the failure rate dropped to zero over 30 days. Lesson: intermittent bugs almost always have a concurrency or state management root cause.

Tell me about mentoring a junior developer?BEHAVIORAL

Describe your approach to code reviews — giving and receiving?BEHAVIORAL

Giving reviews — my framework:

  • Severity labels: Must Fix (blocker), Should Fix (important), Consider (suggestion). No ambiguity about what is required.
  • Explain why, not just what: Not just 'move SOQL outside the loop' — explain the governor limit consequence that will hit production.
  • Acknowledge what is well done: If someone wrote clean bulkification or good test assertions, say so explicitly.
  • Style is a Consider, not Must Fix: Unless the team has an agreed style guide, personal preferences should never block a PR.

Receiving reviews:

  • Every comment gets a response — Fixed or I disagree because X, let's discuss.
  • Never take feedback personally — the code is reviewed, not me.
  • Disagree with Must Fix? Request a conversation, not a comment thread.
  • After a review, look for patterns in feedback — recurring issues mean something to learn.

How do you stay current with Salesforce releases and new features?BEHAVIORAL

Salesforce releases 3 major updates per year: Spring (February), Summer (June), Winter (October).

  • Release Notes: Read for every release — focus on Apex API changes, LWC new capabilities, Flow enhancements, deprecated features and their retirement timelines.
  • Pre-release orgs: Available 4-6 weeks before each release. Test existing code against the new release before it hits production.
  • Trailhead release modules: 1-2 new modules per release — fastest way to learn what changed.
  • Certification maintenance: Salesforce certs require annual maintenance exams. These force you to learn what changed.
  • Deprecation tracking: Personal timeline — when does Process Builder retire? When does WITH SECURITY_ENFORCED become wrong? Plan migrations proactively.
Interview answer

Say: I stay current through release notes and the Salesforce Developer Blog, and I test major features in sandbox before they reach production. I also track the Process Builder deprecation timeline because we have existing automations that need migration to Flow — I want to stay ahead of forced retirements rather than scramble at the deadline.
