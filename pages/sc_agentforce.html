What is Agentforce — explain the architecture at a senior developer level?AGENTFORCE

Agentforce is Salesforce's autonomous AI agent platform (GA: Fall 2024). Agents can reason, act, and complete multi-step tasks without constant human input. As a developer, you extend agents by building Apex @InvocableMethod actions, Flows, and Prompt Templates.

Key developer skill

The most important new skill: write excellent @InvocableMethod descriptions. The description is what the LLM reads to decide WHEN to call your action. Vague description = wrong action called. Write it as: "Use this action when [intent]. Returns [what]. Requires [input]."

How do you build an Apex Action for Agentforce — what makes a good action description?AGENTFORCE
public class AgentLoanStatusAction {

    public class Request {
        @InvocableVariable(label='Borrower Email' required=true
            description='Email address of the borrower asking about their loan')
        public String borrowerEmail;
    }

    public class Response {
        @InvocableVariable(label='Status Summary')
        public String statusSummary;
        @InvocableVariable(label='Next Steps')
        public String nextSteps;
        @InvocableVariable(label='Found')
        public Boolean found;
    }

    // THE DESCRIPTION IS CRITICAL — this is what the LLM reads to decide WHEN to call this
    @InvocableMethod(
        label='Get Loan Status for Borrower'
        description='Use this action when a borrower asks about their loan status, application progress, what stage their loan is in, or what happens next. This action looks up the borrower''s active loan by email and returns the current status and next steps. Requires the borrower''s email address.'
        category='Loan Processing'
    )
    public static List<Response> getLoanStatus(List<Request> requests) {
        List<Response> responses = new List<Response>();
        for(Request req : requests) {
            Response resp = new Response();
            List<Loan__c> loans = [
                SELECT Id, Name, Status__c, Next_Steps__c, LoanOfficer__r.Name
                FROM Loan__c
                WHERE BorrowerEmail__c = :req.borrowerEmail
                AND Status__c NOT IN ('Closed', 'Denied')
                WITH USER_MODE
                ORDER BY CreatedDate DESC LIMIT 1
            ];
            if(loans.isEmpty()) {
                resp.found = false;
                resp.statusSummary = 'No active loan application found for this email address.';
                resp.nextSteps = 'Please contact your loan officer or call our main number.';
            } else {
                Loan__c l = loans[0];
                resp.found = true;
                resp.statusSummary = 'Your loan ' + l.Name + ' is currently in ' + l.Status__c + ' status. Your loan officer is ' + l.LoanOfficer__r.Name + '.';
                resp.nextSteps = String.isNotBlank(l.Next_Steps__c) ? l.Next_Steps__c : 'Your loan officer will contact you with the next steps shortly.';
            }
            responses.add(resp);
        }
        return responses;
    }
}
What is Grounding in Agentforce — how does it access real CRM data and why is it important?AGENTFORCE

Grounding connects the LLM to real, current data. Without grounding, the LLM only knows what was in its training data — potentially outdated or wrong for your business. Grounding makes responses factually accurate.

  • CRM Grounding: Agent calls a Standard CRM Action or Apex Action that queries Salesforce records. The result is returned as context the LLM reads before generating its response
  • Knowledge Article Grounding: Agent searches your Salesforce Knowledge Base via vector search. Retrieved articles become context for answering policy/FAQ questions accurately
  • Data Cloud Grounding: Agent accesses unified customer profiles and calculated insights from Data Cloud via semantic vector search (RAG — Retrieval Augmented Generation)
  • External Grounding via Apex: Apex Action calls an external API (e.g., FRED for mortgage rates) and returns the data as a string the LLM uses to ground its response in real-time data
Why grounding matters in mortgage

Without grounding, if a borrower asks "What is the current 30-year fixed rate?" the LLM might answer with its training data from 2023. With Apex grounding calling the FRED API, the agent answers with today's actual rate. Grounding is what makes AI responses trustworthy in a regulated financial services context.

What is the Einstein Trust Layer and why is it essential for financial services?AGENTFORCE
  • Zero Data Retention: External LLM providers (OpenAI, Anthropic) are contractually prevented from storing or training on Salesforce customer data. Data is used for one response only
  • PII Masking: SSN, credit card numbers, and other PII are automatically detected and replaced with tokens before reaching the LLM. The LLM never sees raw borrower PII
  • Toxicity Detection: LLM output is scanned before shown to users. Harmful content is blocked
  • Audit Trail: All AI interactions are logged in Salesforce for compliance review. For ECOA/FCRA regulated environments, this audit trail is required
  • Data Residency: Data stays within configured geographic boundaries — EU data doesn't leave EU

What are the current limitations of Agentforce a developer should know for interviews?AGENTFORCE
  • Response latency: Agent reasoning + grounding + LLM generation adds 2–10 seconds. Not suitable for real-time sub-second interactions
  • Non-deterministic: Same input may produce slightly different outputs. Critical business logic must remain in deterministic Apex/Flow — not in LLM reasoning
  • Hallucination risk: Even with grounding, LLMs can occasionally produce incorrect information. Always validate critical outputs
  • Context window limits: Very long conversations or large grounded documents may exceed LLM context window, causing truncation
  • No direct DML: The LLM itself doesn't write to Salesforce. It calls Actions, which then perform DML. This adds a step but maintains data integrity
  • Einstein AI credits: High-volume deployments need capacity planning — each agent interaction consumes credits
  • Testing challenges: Non-deterministic behavior means more end-to-end testing needed. Test Apex actions independently (they're just InvocableMethods) and test full agent flows in Agent Builder sandbox
How does Agentforce determine which action to call — explain topics and instructions?AGENTFORCE SCENARIOS

Agent Topics are categories of user intent. Each topic has natural language instructions telling the agent when to handle that type of request.

  • Topic — 'Loan Status Inquiries'. Instructions: 'Handle when user asks about loan status, loan progress, what stage their loan is in, or when their loan will close.'
  • Agent Actions within a Topic — GetLoanStatus, GetEstimatedCloseDate, GetRequiredDocuments. The agent picks the right action based on the user's specific ask.
  • Topic instructions format — natural language, 1-2 sentences per instruction. Tell the agent what the topic covers, what it should NOT handle (hand-off triggers), and how to behave.
  • Example boundary instruction — 'If the user asks about rates, interest calculations, or pre-approval decisions, escalate to a human loan officer rather than answering.'
  • Tone instructions — 'Always address the user by first name. Be empathetic when delivering declined or on-hold status. Never speculate about approval likelihood.'
Good topic design

Well-defined topics with clear boundaries prevent hallucination. If the agent knows exactly what it should and should not handle, it stays within its lane and escalates appropriately rather than guessing.

How do you test and evaluate an Agentforce agent before going to production?AGENTFORCE SCENARIOS
  • Agent Builder Testing panel — in Setup → Agent Builder, test conversations directly. See which topic was selected, which action was called, and what data was returned.
  • Test conversation transcripts — run 20+ realistic test conversations covering expected user inputs, edge cases, and out-of-scope queries. Verify the agent handles each correctly.
  • Boundary testing — deliberately ask questions outside the agent's topics. Verify it escalates rather than hallucinating an answer.
  • Action testing — for each Apex Action, verify: correct data returned, handles 'not found' gracefully, FLS respected (agent running as correct user context), no SOQL/DML errors.
  • User acceptance testing — deploy to a sandbox, have actual loan officers test realistic questions. Their natural language will be different from what the developer tested.
  • Monitoring in production — use Conversation Logs (Setup → Bot Conversation Logs) to review real conversations. Identify patterns where the agent fails and refine topics/instructions.
What is Einstein Copilot for Developers (Code Generation) — how does it work?AGENTFORCE SCENARIOS

Einstein for Developers is an AI coding assistant built into VS Code (Salesforce Extension Pack) and the Developer Console. It generates Apex, LWC, SOQL, and test code from natural language.

  • Generate Apex from comment — write a comment describing what you want, press a shortcut, get code. Example: /* Create a method that finds all loans past due date and updates status to Late */
  • Explain code — select code → ask Einstein to explain it. Useful for understanding inherited code.
  • Generate test class — select an Apex class → ask Einstein to generate a test class with positive, negative, and bulk test cases.
  • Fix errors — paste a compile error → Einstein suggests the fix.
  • SOQL generation — describe what you want to query in plain English → get SOQL.
Always review AI-generated code

Einstein-generated code may have security issues (missing WITH SECURITY_ENFORCED), performance issues (SOQL in loops), or logic errors. Treat it as a starting draft, not a finished solution. Run code review and tests as normal.

How would you design an Agentforce agent for a mortgage company — complete architecture?AGENTFORCE SCENARIOS

Business requirement: Borrowers need self-service access to loan status, document requirements, and appointment scheduling — currently handled by 3 loan officers spending 2 hours each per day on these repetitive queries.

Agent design:

  • Agent persona — 'Mortgage Assistant'. Friendly, professional, empathetic. Always addresses borrower by first name.
  • Topic 1: Loan Status — Action: GetLoanStatus (Apex). Input: loan number or borrower name. Output: status, stage, next steps, estimated close date.
  • Topic 2: Document Requirements — Action: GetRequiredDocuments (Apex). Input: loan ID. Output: list of required documents, which are received, which are outstanding.
  • Topic 3: Appointment Scheduling — Action: GetAvailableSlots + ScheduleAppointment (Apex). Integrates with loan officer calendar.
  • Topic 4: Out of Scope — Escalation topic. Any question about rates, approval decisions, legal advice → 'I'll connect you with your loan officer for that.'
  • Channel — Messaging for In-App and Web (Experience Cloud portal), SMS via Salesforce Messaging.
  • Grounding — Agent queries Loan__c, Borrower__c, Loan_Document__c via Apex Actions. No hallucination risk on factual data.
ROI framing for interview

Frame the design in business terms: 'This agent handles 80% of routine borrower inquiries, freeing loan officers to focus on complex cases and new applications — reducing cost per loan and improving borrower NPS.'
