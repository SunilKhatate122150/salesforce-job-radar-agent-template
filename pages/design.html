Design Patterns in Apex
Senior devs are expected to write maintainable, scalable code using established patterns
Explain the Strategy Pattern in Apex — show code with a real Salesforce example PATTERNS

The Strategy Pattern defines a family of algorithms, encapsulates each one, and makes them interchangeable. The client (caller) works with an interface and doesn't know which concrete implementation it's using.

// Real example: Different loan types have different approval algorithms

// 1. Interface defines the contract
public interface ILoanApprovalStrategy {
    Boolean isApproved(Loan__c loan, CreditReport creditReport);
    String getRejectionReason();
}

// 2. Concrete strategies - one per loan type
public class ConventionalLoanStrategy implements ILoanApprovalStrategy {
    public Boolean isApproved(Loan__c loan, CreditReport cr) {
        return cr.ficoScore >= 620 && loan.DTI__c <= 43 && loan.LTV__c <= 80;
    }
    public String getRejectionReason() { return 'Conventional: FICO min 620, DTI max 43%, LTV max 80%'; }
}

public class FHALoanStrategy implements ILoanApprovalStrategy {
    public Boolean isApproved(Loan__c loan, CreditReport cr) {
        return cr.ficoScore >= 580 && loan.DTI__c <= 57 && loan.LTV__c <= 96.5;
    }
    public String getRejectionReason() { return 'FHA: FICO min 580, DTI max 57%, LTV max 96.5%'; }
}

// 3. Factory to create the right strategy (often combined with Strategy)
public class LoanStrategyFactory {
    public static ILoanApprovalStrategy getStrategy(String loanType) {
        switch on loanType {
            when 'Conventional' { return new ConventionalLoanStrategy(); }
            when 'FHA'          { return new FHALoanStrategy(); }
            when else          { throw new IllegalArgumentException('Unknown loan type: ' + loanType); }
        }
    }
}

// 4. Client code - doesn't know which strategy is used
public class LoanApprovalService {
    public static void processApprovals(List<Loan__c> loans) {
        for(Loan__c loan : loans) {
            ILoanApprovalStrategy strategy = LoanStrategyFactory.getStrategy(loan.LoanType__c);
            CreditReport cr = ExperianService.getCreditReport(loan.BorrowerId__c);
            
            if(strategy.isApproved(loan, cr)) {
                loan.Status__c = 'Approved';
            } else {
                loan.Status__c = 'Denied';
                loan.DenialReason__c = strategy.getRejectionReason();
            }
        }
        update loans;
    }
}
Why this is powerful

To add a new loan type (VA loan, USDA loan), you only create a new class implementing ILoanApprovalStrategy. You don't touch the existing service code at all — Open/Closed Principle. This is what separates senior dev thinking from mid-level.

What is the Service Layer pattern — why is it the most important pattern in Salesforce?PATTERNS

Service Layer is where business logic lives. Without it, logic gets duplicated across triggers, LWC controllers, batch jobs, and APIs.

// WITHOUT service layer — same logic duplicated everywhere
// In trigger after update: sendApprovalEmail(loan); updateBorrower(loan);
// In LWC @AuraEnabled method: same calls duplicated!
// In batch finish: same calls duplicated again!

// WITH service layer — one place, called from anywhere
public with sharing class LoanApprovalService {
  public static void approve(List<Loan__c> loans) {
    List<Loan__c> toUpdate = new List<Loan__c>();
    for (Loan__c l : loans) {
      l.Status__c = 'Approved'; l.Approval_Date__c = Date.today(); toUpdate.add(l);
    }
    update toUpdate;
    LoanEmailService.sendApprovalEmails(toUpdate);
    BorrowerService.updateActiveLoanCounts(toUpdate);
  }
}
// Trigger calls: LoanApprovalService.approve(changedLoans);
// LWC @AuraEnabled calls: LoanApprovalService.approve(new List<Loan__c>{loan});
// Batch calls: LoanApprovalService.approve(scope); — SAME service, no duplication
What is the Trigger Handler pattern — complete implementation with framework?PATTERNS
// TriggerHandler.cls — virtual base (framework layer)
public virtual class TriggerHandler {
  public virtual void onBeforeInsert() {} public virtual void onAfterInsert() {}
  public virtual void onBeforeUpdate() {} public virtual void onAfterUpdate() {}
  public virtual void onBeforeDelete() {} public virtual void onAfterDelete() {}
  public void run() {
    if (Trigger.isBefore) {
      if (Trigger.isInsert) onBeforeInsert();
      if (Trigger.isUpdate) onBeforeUpdate();
      if (Trigger.isDelete) onBeforeDelete();
    } else {
      if (Trigger.isInsert) onAfterInsert();
      if (Trigger.isUpdate) onAfterUpdate();
      if (Trigger.isDelete) onAfterDelete();
    }
  }
}

// LoanTriggerHandler.cls — domain-specific logic
public class LoanTriggerHandler extends TriggerHandler {
  private List<Loan__c> newLoans  = (List<Loan__c>) Trigger.new;
  private Map<Id,Loan__c> oldMap  = (Map<Id,Loan__c>) Trigger.oldMap;
  public override void onBeforeInsert() { LoanValidationService.validate(newLoans); }
  public override void onAfterUpdate()  { LoanStatusService.handleChange(newLoans, oldMap); }
}

// LoanTrigger.trigger — one line
trigger LoanTrigger on Loan__c (before insert, before update, after insert, after update, after delete) {
  new LoanTriggerHandler().run();
}
What is the Strategy pattern — complete Salesforce implementation?PATTERNS
public interface ILoanProcessor {
  void validate(Loan__c loan);
  Decimal calculateRate(Loan__c loan);
  List<String> requiredDocs(Loan__c loan);
}
public class ConventionalProcessor implements ILoanProcessor {
  public void validate(Loan__c l)  { if (l.Credit_Score__c < 620) l.addError('Need 620+ score'); }
  public Decimal calculateRate(Loan__c l) { return 6.5 + (l.Credit_Score__c < 740 ? 0.5 : 0.0); }
  public List<String> requiredDocs(Loan__c l) { return new List<String>{'W2','Tax Return','Bank Statement'}; }
}
public class FHAProcessor implements ILoanProcessor {
  public void validate(Loan__c l)  { if (l.Credit_Score__c < 580) l.addError('FHA needs 580+ score'); }
  public Decimal calculateRate(Loan__c l) { return 6.25; }
  public List<String> requiredDocs(Loan__c l) { return new List<String>{'W2','FHA Certificate'}; }
}
// Factory + Strategy combined
public class LoanProcessorFactory {
  static Map<String,ILoanProcessor> MAP = new Map<String,ILoanProcessor>{
    'Conventional'=>new ConventionalProcessor(), 'FHA'=>new FHAProcessor()
  };
  public static ILoanProcessor get(String type) {
    if (!MAP.containsKey(type)) throw new LoanException('Unknown type: '+type);
    return MAP.get(type);
  }
}
// Usage — caller never knows which processor it's using
ILoanProcessor proc = LoanProcessorFactory.get(loan.Loan_Type__c);
proc.validate(loan);
loan.Interest_Rate__c = proc.calculateRate(loan);
What is the Observer pattern — how does it map to Platform Events in Salesforce?PATTERNS

Observer defines a one-to-many relationship: when subject changes, all observers are notified. Platform Events are the native Salesforce implementation.

// Subject (publisher) — just publishes. Knows nothing about observers.
trigger LoanTrigger on Loan__c (after update) {
  List<Loan_Event__e> events = new List<Loan_Event__e>();
  for (Loan__c l : Trigger.new)
    if (l.Status__c != Trigger.oldMap.get(l.Id).Status__c)
      events.add(new Loan_Event__e(Loan_Id__c=l.Id, Status__c=l.Status__c));
  EventBus.publish(events);
}

// Observer 1 — email notifications (independent, can be deployed/removed independently)
trigger EmailObserver on Loan_Event__e (after insert) { LoanEmailService.process(Trigger.new); }

// Observer 2 — LOS sync (independent)
trigger LOSObserver on Loan_Event__e (after insert) {
  for (Loan_Event__e e : Trigger.new) System.enqueueJob(new LOSSyncQueueable(e.Loan_Id__c));
}
// Observer 3 — analytics (Flow subscriber — no code at all)
// Adding a new observer requires ZERO changes to the trigger (open/closed principle)
What is the Decorator pattern in Apex — when is it useful?PATTERNS
public interface IEmailService { void sendApproval(Loan__c loan); }

// Base
public class BasicEmailService implements IEmailService {
  public void sendApproval(Loan__c loan) { /* send basic email */ }
}
// Decorator 1 — adds logging without changing base
public class LoggingEmailDecorator implements IEmailService {
  private IEmailService inner;
  public LoggingEmailDecorator(IEmailService s) { inner = s; }
  public void sendApproval(Loan__c loan) {
    insert new Email_Log__c(Loan__c=loan.Id, Sent_At__c=DateTime.now());
    inner.sendApproval(loan); // delegate
  }
}
// Decorator 2 — adds CC to loan officer
public class LoanOfficerCCDecorator implements IEmailService {
  private IEmailService inner;
  public LoanOfficerCCDecorator(IEmailService s) { inner = s; }
  public void sendApproval(Loan__c loan) {
    inner.sendApproval(loan);
    sendLOCopy(loan); // additional behaviour
  }
}
// Compose: wraps base with multiple decorators
IEmailService service = new LoanOfficerCCDecorator(new LoggingEmailDecorator(new BasicEmailService()));
service.sendApproval(loan); // all three layers execute
What is the Facade pattern — how does it simplify complex Salesforce orchestration?PATTERNS
// WITHOUT Facade — LWC controller knows too much about internals:
// CreditService.run(); OfacService.run(); DocService.generate(); NotifService.send(); LOSService.sync();

// WITH Facade — single clean method hides all complexity
public class LoanApplicationFacade {
  @AuraEnabled
  public static SubmitResult submit(Id loanId) {
    try {
      CreditService.runCreditCheck(loanId);
      OfacService.runOfacCheck(loanId);
      DocumentService.generateDocs(loanId);
      NotificationService.notifyUnderwriter(loanId);
      System.enqueueJob(new LOSSyncQueueable(loanId));
      AuditService.log('LOAN_SUBMITTED', loanId);
      return new SubmitResult(true, 'Application submitted');
    } catch (Exception e) { return new SubmitResult(false, e.getMessage()); }
  }
}
// LWC calls: await submitLoan({ loanId }); — one call, complete orchestration hidden
What is the Singleton pattern in Apex — when does it save governor limits?PATTERNS
// Without Singleton — each call to getConfig() runs a SOQL query
// In a trigger processing 200 records: 200 calls = 200 SOQL — hits the limit!

// With Singleton — query happens once, result cached for the transaction
public class LoanConfigCache {
  private static LoanConfigCache instance;
  private Map<String, Loan_Config__mdt> cache;

  private LoanConfigCache() { // private — nobody can call new LoanConfigCache()
    cache = new Map<String, Loan_Config__mdt>();
    for (Loan_Config__mdt cfg : [SELECT Loan_Type__c, Max_Amount__c, Base_Rate__c FROM Loan_Config__mdt])
      cache.put(cfg.Loan_Type__c, cfg);
  }

  public static LoanConfigCache getInstance() {
    if (instance == null) instance = new LoanConfigCache(); // 1 SOQL total
    return instance;
  }

  public Loan_Config__mdt getConfig(String loanType) { return cache.get(loanType); }
}

// Trigger handler — 200 records, 1 SOQL total
for (Loan__c loan : Trigger.new) {
  Loan_Config__mdt cfg = LoanConfigCache.getInstance().getConfig(loan.Loan_Type__c);
  loan.Max_Rate__c = cfg.Base_Rate__c + 2.0;
}
What is the Unit of Work pattern — how does it reduce DML statements?PATTERNS

Unit of Work collects all pending DML operations across a business process and executes them in one coordinated batch at the end — minimising DML statements and ensuring transactional integrity.

// Without UoW — scattered DML (3 separate statements)
public static void approveLoan(Id loanId) {
  Loan__c l = [SELECT Id FROM Loan__c WHERE Id=:loanId];
  l.Status__c='Approved'; update l; // DML 1
  Account a = [SELECT Id, Active_Loans__c FROM Account WHERE Id=:l.Account__c];
  a.Active_Loans__c++; update a; // DML 2
  insert new Approval_Log__c(Loan__c=l.Id); // DML 3
}

// With UoW pattern (manual implementation without fflib)
public class UnitOfWork {
  List<SObject> toInsert = new List<SObject>();
  List<SObject> toUpdate = new List<SObject>();
  List<SObject> toDelete = new List<SObject>();
  public void registerNew(SObject obj)    { toInsert.add(obj); }
  public void registerDirty(SObject obj)  { toUpdate.add(obj); }
  public void registerDeleted(SObject obj){ toDelete.add(obj); }
  public void commitWork() {
    if (!toInsert.isEmpty()) insert toInsert; // 1 DML
    if (!toUpdate.isEmpty()) update toUpdate; // 1 DML
    if (!toDelete.isEmpty()) delete toDelete; // 1 DML
  }
}
UnitOfWork uow = new UnitOfWork();
uow.registerDirty(loan); uow.registerDirty(account); uow.registerNew(approvalLog);
uow.commitWork(); // 2 DML statements instead of 3
What is the Null Object pattern — how does it eliminate defensive null checks?PATTERNS
// WITHOUT null object — null checks everywhere
ILoanProcessor proc = LoanProcessorFactory.get(loan.Loan_Type__c); // might return null
if (proc != null) { proc.validate(loan); proc.calculateRate(loan); }
else { /* default handling duplicated everywhere */ }

// WITH null object — factory NEVER returns null
public class NullLoanProcessor implements ILoanProcessor {
  public void validate(Loan__c l) {} // no-op — does nothing safely
  public Decimal calculateRate(Loan__c l) { return 0; }
  public List<String> requiredDocs(Loan__c l) { return new List<String>(); }
}
public class LoanProcessorFactory {
  public static ILoanProcessor get(String type) {
    Map<String,ILoanProcessor> m = getMap();
    return m.containsKey(type) ? m.get(type) : new NullLoanProcessor(); // never null
  }
}
// Calling code — no null check needed ever
ILoanProcessor proc = LoanProcessorFactory.get(loan.Loan_Type__c);
proc.validate(loan); // safe — worst case is a no-op
What is Separation of Concerns — how do you apply it across Apex layers?PATTERNS

Every class should have one reason to change. In Salesforce, the layers are:

  • Trigger — dispatcher only. One line: new Handler().run(). No business logic.
  • Trigger Handler — routes trigger events to the right service methods. No SOQL, no business rules.
  • Service Layer — business logic. Orchestrates domain operations. Called from trigger handler, LWC controller, batch, API.
  • Selector Layer — all SOQL queries for an object. No business logic, only queries.
  • Domain Layer — SObject-specific business rules (validation, defaults, derivations).
  • Integration Layer — HTTP callouts, JSON mapping, external system communication.
  • Controller Layer — @AuraEnabled methods. Thin — calls Service layer, returns results to LWC.
Why it matters

When Loan approval logic changes, you change LoanApprovalService only. Not the trigger, not the LWC controller, not the batch. Without SoC, one business rule change requires hunting through 10 files.
