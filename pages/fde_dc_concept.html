FDE Prep — Data Cloud Core Concepts
8 questions covering the data lifecycle, identity resolution, segments, and architecture
Walk through the complete data lifecycle in Data Cloud from ingestion to activation.CONCEPT▼
  • Step 1 — Ingestion: Data enters via connectors (S3 batch, CRM direct, Streaming API, Web SDK) and lands in Data Lake Objects (DLO)
  • Step 2 — Mapping: DLO fields mapped to standard or custom Data Model Objects (DMO) — harmonizing data across sources
  • Step 3 — Identity Resolution: Match rules stitch records from different DMOs into a single Unified Individual profile
  • Step 4 — Calculated Insights: Aggregated metrics computed on unified data (total loan count, LTV)
  • Step 5 — Segmentation: Build audience segments using Segment Builder — filter by attributes, events, insights
  • Step 6 — Activation: Push segments to Marketing Cloud, ad platforms, or back to Salesforce CRM
What is a Unified Individual and how is it created?CONCEPT▼
  • A Unified Individual is the master 360° record representing one real person, assembled from multiple source records across systems
  • Created by Identity Resolution — configurable match rules (exact email, fuzzy name+phone, loyalty ID, cookie ID)
  • Reconciliation Rules decide which field value wins when sources conflict — most recent, most frequent, or source priority
  • Each source record remains intact as a "party" linked to the Unified Individual — non-destructive
  • Example: CRM Contact + LOS borrower + web session user → all linked to one Unified Individual
Emphasis point

Stress the non-destructive nature — source data is preserved, unified profile is a virtual construct built on top.

What are Calculated Insights? How do you create them?CONCEPT▼
  • Calculated Insights are metric computations run on unified Data Cloud data using SQL-style expressions
  • They produce: total mortgage balance, number of active loans, last login date, lifetime payment total
  • Created in: Data Cloud Setup → Calculated Insights → New Insight; select DMO, define expression, set refresh schedule
  • Outputs stored on Unified Individual — available in Segment Builder and CRM records via sync
  • Used in Agentforce grounding — agent retrieves pre-computed insight, avoiding complex SOQL at query time
What is the difference between a Segment and an Activation Target?COMPARE▼
  • Segment — a filtered audience of Unified Individuals (e.g. "ARM borrowers with credit score above 720")
  • Activation Target — the destination system where segment data is pushed (Marketing Cloud, Google Ads, CRM)
  • Build Segment first, then activate to one or more targets
  • Activation Target defines: which fields to send, refresh frequency, mapping to target schema
  • Activation can enrich CRM — stamp "High Refinance Intent" on Contact from a Data Cloud segment
How does Data Cloud relate to a Salesforce CRM org? Same platform?CONCEPT▼
  • No — Data Cloud is a separate product provisioned on top of a Salesforce org but architecturally distinct
  • Runs on Salesforce's Hyperforce infrastructure with its own data lake and compute layer
  • Connected to CRM via Salesforce CRM Connector — native, no-code, syncs standard objects
  • Data Cloud objects (DLO, DMO, Unified Individual) are NOT standard SF objects — not accessible via standard SOQL
  • CRM Sync: enriched insights written back to CRM as standard fields for reports and page layouts
Key insight

"Data Cloud is the customer intelligence layer beneath all Einstein AI features — it's the data platform, not just storage."

What are Data Spaces and why do they matter for enterprise customers?ARCHITECTURE▼
  • Data Spaces = logical partitions within a single Data Cloud org for different business units/brands/regions
  • Use case: financial holding with Mortgage, Insurance, Wealth divisions — each gets isolated Data Space
  • Admins control which users access which Data Space — role-based access
  • Segments, identity resolution, calculated insights are scoped to a Data Space
  • For mortgage: FCRA requires data access controls; Data Spaces enforce segregation of sensitive financial data
Explain Match Rules vs Reconciliation Rules with a concrete example.SCENARIO▼

Match Rules = determine WHICH records belong to the same person. Reconciliation Rules = determine WHICH version of the truth wins when matched records conflict.

Source 1 (CRM): John Smith, john@gmail.com, 123 Main St (Jan 2026) Source 2 (LOS): Jonathan Smith, john@gmail.com, 456 Oak Ave (Mar 2026) Source 3 (Web): J. Smith, john@gmail.com, (no address) Match Rule: Exact email → All 3 linked to same Unified Individual Reconciliation: Name → Source Priority (CRM wins) → "John Smith" Address → Most Recent (LOS wins) → "456 Oak Ave" Email → Same across all → "john@gmail.com"
Quick summary

Match = "are these the same person?" Reconciliation = "which version of the truth do we use?"
