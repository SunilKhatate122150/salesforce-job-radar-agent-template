What are Platform Events? How do you publish and subscribe? Show complete code. PLATFORM EVENTS

Platform Events are a messaging solution built on Salesforce's streaming infrastructure. They enable event-driven architecture — loosely coupled components communicate through events rather than direct calls. The publisher doesn't know who the subscribers are.

// 1. Define Platform Event: Loan_Status_Update__e
// Fields: LoanId__c (Text), NewStatus__c (Text), UpdatedBy__c (Text), Timestamp__c (DateTime)

// 2. PUBLISH from Apex (trigger or service class)
public class LoanStatusEventPublisher {
    
    public static void publishStatusChange(List<Loan__c> updatedLoans) {
        List<Loan_Status_Update__e> events = new List<Loan_Status_Update__e>();
        
        for(Loan__c loan : updatedLoans) {
            events.add(new Loan_Status_Update__e(
                LoanId__c = loan.Id,
                NewStatus__c = loan.Status__c,
                UpdatedBy__c = UserInfo.getUserId(),
                Timestamp__c = DateTime.now()
            ));
        }
        
        // Publish and check results
        List<Database.SaveResult> results = EventBus.publish(events);
        for(Database.SaveResult sr : results) {
            if(!sr.isSuccess()) {
                System.debug('Event publish failed: ' + sr.getErrors()[0].getMessage());
            }
        }
    }
}

// 3. SUBSCRIBE via Apex trigger
trigger LoanStatusUpdateTrigger on Loan_Status_Update__e (after insert) {
    List<Loan_Status_Update__e> events = Trigger.new;
    // Trigger.new contains the events - NOT database records
    
    List<Id> loanIds = new List<Id>();
    for(Loan_Status_Update__e event : events) {
        loanIds.add(event.LoanId__c);
    }
    
    // Now do whatever needs to happen when loan status changes
    // e.g., notify DocuSign, update related records, send email
    DocuSignNotificationService.notifyOnStatusChange(loanIds, events);
}

// 4. Publish Behavior: IMPORTANT DISTINCTION
// PublishImmediately: event fires even if transaction ROLLS BACK
// PublishAfterCommit (default): event fires only if transaction COMMITS successfully
// For financial systems: ALWAYS use PublishAfterCommit (default)
Your project story

"In our Mortgage Origination Platform, we used Platform Events to decouple the loan status update process from DocuSign document generation. When a loan moved to 'Approved' status, a Platform Event was published. A separate trigger subscribed to that event and triggered the DocuSign envelope. This meant our loan processing code didn't need to know about DocuSign — clean separation of concerns."

// Subscribe to Loan__c CDC changes in Apex trigger
trigger LoanCDCTrigger on LoanChangeEvent (after insert) {
  for (LoanChangeEvent evt : Trigger.new) {
    String changeType = evt.ChangeEventHeader.changeType; // CREATE UPDATE DELETE UNDELETE
    List<String> changedFields = evt.ChangeEventHeader.changedFields;
    List<Id> recordIds = evt.ChangeEventHeader.recordIds;

    if (changeType == 'UPDATE' && changedFields.contains('Status__c')) {
      System.enqueueJob(new StatusSyncQueueable(recordIds[0], evt.Status__c));
    }
  }
}
What is the ReplayId in Platform Events — how does event replay work?PLATFORM EVENTS

Every Platform Event gets a monotonically increasing ReplayId. Subscribers use it to resume from a specific point in the 72-hour retention window — critical for disaster recovery.

// empApi in LWC — replay options
// -1 = new events only (default)
// -2 = replay ALL retained events from earliest available
// specific number = replay from that event onward

import { subscribe } from 'lightning/empApi';
subscribe('/event/Loan_Status_Change__e', -2, callback); // replay all retained
subscribe('/event/Loan_Status_Change__e', 12345, callback); // replay from event 12345

// Pattern: store last processed ReplayId in Custom Setting
// On system restart: retrieve stored ReplayId ΓåÆ subscribe from that point
// Ensures no events are missed during downtime (within 72-hour window)
Disaster recovery

If your external system is down for 2 hours it can resubscribe with the last processed ReplayId and receive every missed event. This is the killer advantage of Platform Events over custom polling tables.

// Trigger with Publish After Commit PE (correct for data-driven events)
trigger LoanTrigger on Loan__c (after update) {
  List<Loan_Status_Change__e> events = new List<Loan_Status_Change__e>();
  for (Loan__c loan : Trigger.new) {
    if (loan.Status__c != Trigger.oldMap.get(loan.Id).Status__c)
      events.add(new Loan_Status_Change__e(Loan_Id__c=loan.Id, New_Status__c=loan.Status__c));
  }
  EventBus.publish(events); // fires ONLY if update commits — no phantom events
}
How do you subscribe to Platform Events in LWC using empApi?PLATFORM EVENTS
import { LightningElement } from 'lwc';
import { subscribe, unsubscribe, onError } from 'lightning/empApi';

export default class LoanStatusMonitor extends LightningElement {
  subscription = {};

  async connectedCallback() {
    // Subscribe — -1 means new events only
    this.subscription = await subscribe('/event/Loan_Status_Change__e', -1, (response) => {
      const payload = response.data.payload;
      this.handleUpdate(payload.Loan_Id__c, payload.New_Status__c);
    });

    onError(error => console.error('Streaming error:', error));
  }

  handleUpdate(loanId, status) {
    // Update UI in real time — no page refresh
    this.dispatchEvent(new CustomEvent('statuschange', { detail: { loanId, status } }));
  }

  disconnectedCallback() {
    unsubscribe(this.subscription, response => console.log('Unsubscribed:', response));
  }
}

How do you use Platform Events to decouple a trigger from complex post-processing?PLATFORM EVENTS

Classic pattern: trigger publishes a Platform Event ΓåÆ separate subscriber trigger handles downstream work. The original trigger stays thin and fast.

// Thin trigger — just publishes, no logic
trigger LoanTrigger on Loan__c (after update) {
  List<Loan_Event__e> evts = new List<Loan_Event__e>();
  for (Loan__c l : Trigger.new) {
    if (l.Status__c != Trigger.oldMap.get(l.Id).Status__c)
      evts.add(new Loan_Event__e(Loan_Id__c=l.Id, Status__c=l.Status__c));
  }
  EventBus.publish(evts);
}

// Email observer trigger — independent, can be deployed/removed without touching loan trigger
trigger LoanEmailObserver on Loan_Event__e (after insert) {
  LoanEmailService.processEvents(Trigger.new);
}

// LOS sync observer trigger — another independent subscriber
trigger LoanLOSObserver on Loan_Event__e (after insert) {
  for (Loan_Event__e e : Trigger.new)
    System.enqueueJob(new LOSSyncQueueable(e.Loan_Id__c, e.Status__c));
}
What is the Streaming API — what are all its event types?PLATFORM EVENTS
  • PushTopic Events (legacy) — define a SOQL query, subscribers notified when matching records change. Being replaced by CDC.
  • Generic Streaming Events — lightweight, schema-less JSON messages. No retention. Good for simple real-time notifications.
  • Platform Events — schematized, 72-hour retention, high volume. Subscribe via Apex trigger, Flow, LWC empApi, or external CometD client.
  • Change Data Capture (CDC) — auto-generated per object. 3-day retention. Best for external system data sync.
  • Monitoring Events (Event Monitoring) — read-only. LoginEvent, ApiEvent, ListViewEvent for security audit/monitoring.
